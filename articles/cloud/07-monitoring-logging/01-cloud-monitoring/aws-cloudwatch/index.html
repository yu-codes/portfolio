概述

  • 用於收集並追蹤 AWS 資源的指標、監控日誌與設定警示
  • 支援服務：EC2、ALB、S3、Lambda、DynamoDB、RDS 等

指標（Metrics）

解析度

類型間隔費用
標準每分鐘免費
高解析度最多 1 秒付費

EC2 預設指標

支援不支援（需 Agent）
CPU 使用率記憶體使用量
網路 I/O磁碟使用量
磁碟 I/O-

CloudWatch Logs

功能

  • 收集應用程式日誌
  • 即時日誌搜尋
  • 日誌保留設定
  • 匯出至 S3

CloudWatch Logs Agent

  • 安裝於 EC2
  • 收集自訂指標與應用程式日誌

權限需求

任一服務若要寫入 CloudWatch Logs，需具備：

  • logs:CreateLogGroup
  • logs:CreateLogStream
  • logs:PutLogEvents

CloudWatch Alarms

狀態

狀態說明
OK指標在閾值內
ALARM指標超過閾值
INSUFFICIENT_DATA資料不足

警示動作

  • 發送 SNS 通知
  • 自動擴展動作
  • EC2 動作（停止、終止、重啟、復原）

EC2 復原

  • 可根據警示自動復原 EC2 實例
  • 保留實例 ID、IP、EBS 等

CloudWatch Dashboard

  • 視覺化圖表
  • 支援跨帳號、跨區域
  • 可自訂時間範圍

CloudWatch Events / EventBridge

  • 排程任務（類似 cron）
  • 回應 AWS 服務事件
  • 觸發 Lambda、SNS 等目標

自訂指標

發布自訂指標

  • 使用 PutMetricData API
  • 可設定維度（Dimensions）
  • 支援高解析度（1 秒）

應用範例

  • 記憶體使用率
  • 應用程式效能指標
  • 業務指標

Metric Math

  • 對指標進行數學運算
  • 建立複合指標
  • 用於警示和儀表板

Logs Insights

  • 互動式日誌查詢
  • 類 SQL 語法
  • 視覺化查詢結果

CloudWatch 進階功能

Contributor Insights

  • 識別高流量貢獻者
  • 找出效能問題的根本原因
  • 支援自訂規則

CloudWatch Synthetics

  • 建立 Canary 監控端點和 API
  • 模擬使用者行為
  • 主動發現問題

CloudWatch Internet Monitor

  • 監控網際網路效能
  • 影響應用程式可用性的洞察
  • 識別網路問題

CloudWatch Application Signals

  • 應用程式效能監控（APM）
  • 自動化儀表板
  • 服務層級目標（SLO）追蹤

最佳實踐

  1. 安裝 CloudWatch Agent：收集更多指標
  2. 設定適當的警示：及早發現問題
  3. 使用 Dashboard：視覺化監控
  4. 配置日誌保留：控制成本
  5. 使用 Logs Insights：分析日誌
  6. 使用 Synthetics：主動監控端點
  7. 啟用 Contributor Insights：識別異常流量

CloudWatch 架構深入解析

概述

  • AWS 的核心資料收集服務
  • 收集和管理操作資料
  • 三個主要產品：Metrics、Logs、Events

CloudWatch 資料流

AWS 服務/應用程式 → CloudWatch Metrics/Logs → Alarms/Dashboard/Insights

CloudWatch Logs 深入解析

架構

層級說明
Log Group日誌的容器
Log StreamLog Group 內的日誌序列
Log Event單筆日誌記錄（時間戳記 + 訊息）

重要設定

  • 保留設定：從 1 天到永久
  • 加密：使用 KMS 加密
  • 匯出：可匯出到 S3（非即時）

Subscriptions

CloudWatch Logs → Subscription Filter → Lambda/Kinesis/Firehose
  • 即時日誌處理
  • 可跨帳號傳遞

Unified CloudWatch Agent

  • 收集日誌和指標
  • 支援 EC2 和本地伺服器
  • 可收集：
    • 系統層級指標（記憶體、磁碟等）
    • 應用程式日誌
    • 自訂指標

CloudWatch Alarms 深入解析

狀態

狀態說明
OK指標在定義的閾值內
ALARM指標在定義的閾值外
INSUFFICIENT_DATA資料不足以判斷

警示配置

設定說明
Period指標評估的時間長度
Evaluation Periods判斷警示前的評估次數
Datapoints to Alarm觸發警示所需的違規資料點

動作類型

  • SNS 通知
  • Auto Scaling 動作
  • EC2 動作（停止、終止、重啟、恢復）

EventBridge 詳解

概述

  • CloudWatch Events 的進化版
  • 預設事件匯流排 + 自訂事件匯流排
  • 可接收第三方應用程式事件

事件來源

來源範例
AWS 服務EC2 狀態變更、S3 物件建立
排程Cron 表達式或 rate 表達式
第三方SaaS 應用程式
自訂您的應用程式

規則目標

  • Lambda
  • SQS
  • SNS
  • Step Functions
  • Kinesis
  • 更多…

事件模式

{
  "source": ["aws.ec2"],
  "detail-type": ["EC2 Instance State-change Notification"],
  "detail": {
    "state": ["stopped"]
  }
}

AWS Config

概述

  • 記錄 AWS 資源的配置變更
  • 評估資源是否符合合規規則
  • 不阻止變更，只記錄和警示

核心概念

概念說明
Configuration Item資源的配置快照
Configuration History資源的歷史配置
Configuration Stream配置變更的即時串流

Config Rules

類型說明
AWS Managed RulesAWS 預定義的規則
Custom Rules使用 Lambda 自訂

範例規則

  • restricted-ssh：確保安全群組不允許 0.0.0.0/0 SSH
  • encrypted-volumes：確保 EBS 磁碟已加密
  • required-tags：確保資源有必要的標籤

修復

  • Config Rules 可觸發自動修復
  • 使用 SSM Automation Documents

CloudWatch vs CloudTrail vs Config

項目CloudWatchCloudTrailConfig
用途效能監控API 稽核合規性
資料指標、日誌API 呼叫事件資源配置
重點資源做了什麼誰做了什麼資源配置變更
範例CPU 使用率誰修改了 SGSG 規則變更

整合使用案例

偵測安全群組未授權變更：

CloudTrail（記錄 API）
     ↓
CloudWatch Logs（串流）
     ↓
Metric Filter + Alarm
     ↓
SNS 通知安全團隊
     ↓
Config Rule 評估合規性
     ↓
自動修復（SSM）

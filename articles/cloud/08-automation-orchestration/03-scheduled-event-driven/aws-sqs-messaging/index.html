概述

  • 完全託管佇列服務
  • 訊息保留時間：預設 4 天，最長 14 天
  • 單筆訊息最大 256KB
  • 可無限量訊息排隊

Standard SQS 特性

特性說明
吞吐量無限制
發送/接收延遲<10ms
送達保證至少一次（可能重複）
訊息順序盡力維持（可能無序）

消費者行為

  • 消費者（EC2 或 Lambda）以批次方式 poll（最多 10 筆）
  • 處理完後必須手動刪除訊息，否則會再次被處理
  • 需在 SQS 存取政策中允許 Producer 和 Consumer 存取佇列

Message Visibility Timeout

  • 訊息被取出處理期間會變為不可見
  • 避免被其他消費者重複處理
  • 預設 30 秒，可調整 0 秒 ~ 12 小時
  • 處理時間超過 Visibility Timeout 時，訊息會再次可見

死信佇列（DLQ）

  • 將多次處理失敗的訊息轉入另一個 SQS 中保存
  • 設定 maxReceiveCount：訊息被接收次數超過此值後移至 DLQ
  • 適用於錯誤分析和重新處理

訊息延遲

類型設定位置最大延遲
Delivery DelayQueue 層級15 分鐘
DelaySeconds訊息層級15 分鐘
  • 預設延遲為 0 秒

長輪詢（Long Polling）

  • 設定 ReceiveMessageWaitTimeSeconds > 0
  • 當無訊息時等待訊息出現再回傳
  • 可節省費用和減少空請求
  • 建議值：1-20 秒

FIFO 佇列

特性StandardFIFO
訊息順序盡力維持嚴格保證
送達保證至少一次僅一次
吞吐量無限制300/秒（非批次）或 3000/秒（批次）
佇列名稱任意必須以 .fifo 結尾
  • 無法從 Standard 轉為 FIFO

FIFO 關鍵概念

概念說明
Message Group ID同一 Group 內訊息保證順序
Message Deduplication ID防止重複訊息（5 分鐘內）

SQS Temporary Queue Client

  • 實作 SQS 版 Request-Response 模式
  • 適用於需要回應的非同步通訊

與 Auto Scaling 整合

應用範例

使用 CloudWatch 自訂指標觸發 Auto Scaling：

指標 = SQS 佇列長度 / EC2 數量

當指標超過閾值時，增加消費者數量。

與其他服務整合

  • Lambda：作為觸發器自動處理訊息
  • SNS：Fan-out 架構
  • EventBridge：事件驅動架構
  • S3：物件事件通知
  • EC2 Auto Scaling：根據佇列深度擴展

最佳實踐

  1. 使用長輪詢：減少空請求和成本
  2. 配置 DLQ：處理失敗訊息
  3. 適當設定 Visibility Timeout：大於預期處理時間
  4. 使用批次操作：提高效率
  5. FIFO 用於順序需求：嚴格順序場景
  6. 監控佇列深度：設定警示

SQS 架構深入解析

概述

  • 公共服務，在 AWS 公共區域端點存取
  • 完全託管、高可用
  • 用於分離應用程式元件

訊息處理流程

Producer → SQS Queue → Consumer（Poll）→ 處理 → 刪除訊息

重要概念

概念說明
訊息大小最大 256 KB
保留期限1 分鐘 - 14 天（預設 4 天）
可見性超時0 秒 - 12 小時（預設 30 秒）
加密SSE-SQS 或 SSE-KMS

Standard vs FIFO 詳解

Standard 佇列

特性說明
順序盡力維持，但不保證
送達至少一次（可能重複）
效能近乎無限擴展
適用大多數使用案例

FIFO 佇列

特性說明
順序嚴格保證
送達僅一次
效能3,000 msg/s（批次）、300 msg/s（非批次）
命名必須以 .fifo 結尾

FIFO 重要概念

  • Message Group ID：同一群組內訊息按順序處理
  • Message Deduplication ID：5 分鐘內去重

死信佇列（DLQ）詳解

功能

  • 接收多次處理失敗的訊息
  • 設定 maxReceiveCount（超過後移至 DLQ）
  • 用於問題診斷和偵錯

重新驅動

  • 可將 DLQ 訊息移回原始佇列
  • 修復問題後重新處理

設定建議

  • DLQ 類型必須與來源佇列相同（Standard 對 Standard）
  • 設定足夠長的保留期限

Delay Queue

概念

  • 訊息在指定時間後才可見
  • 佇列層級設定：影響所有新訊息
  • 訊息層級設定：DelaySeconds 參數

限制

  • 最大延遲：15 分鐘

使用案例

  • 需要延遲處理的任務
  • 避免立即處理

SQS Extended Client Library

問題

  • SQS 訊息最大 256 KB
  • 某些場景需要更大的訊息

解決方案

  • 使用 SQS Extended Client Library
  • 實際資料存儲在 S3
  • SQS 訊息包含 S3 物件的參照

架構

Producer → S3（大型資料）→ SQS（參照）→ Consumer → 從 S3 讀取

SQS 與 Lambda 整合

事件來源映射

  • Lambda 輪詢 SQS
  • 批次處理訊息
  • 自動刪除成功處理的訊息

配置

設定說明
Batch Size每次調用的訊息數（1-10）
Batch Window等待收集訊息的時間
並發與佇列類型相關

失敗處理

  • 失敗的訊息保持可見
  • 可配置 DLQ 處理失敗訊息

SQS 安全性

加密

類型說明
傳輸中HTTPS 端點
靜態SSE-SQS（預設）或 SSE-KMS

存取控制

  • IAM Policy：誰可以執行 SQS 操作
  • SQS Access Policy：資源政策，控制跨帳號存取

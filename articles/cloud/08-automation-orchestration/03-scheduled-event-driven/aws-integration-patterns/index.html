概述

應用程式整合是現代架構的關鍵，AWS 提供多種服務來實現不同的整合模式。

同步 vs 非同步

同步整合

特性說明
通訊方式請求-回應
耦合程度緊耦合
等待呼叫者等待回應
適用場景即時查詢、API 呼叫

非同步整合

特性說明
通訊方式訊息傳遞
耦合程度鬆耦合
等待呼叫者不等待
適用場景任務處理、事件驅動

常見整合模式

1. Point-to-Point（點對點）

Producer → SQS Queue → Consumer
  • 使用 SQS
  • 一對一通訊
  • 訊息只被一個消費者處理

2. Publish-Subscribe（發布訂閱）

Publisher → SNS Topic → Subscriber 1
                      → Subscriber 2
                      → Subscriber 3
  • 使用 SNS
  • 一對多通訊
  • 所有訂閱者收到相同訊息

3. Fan-out（扇出）

Producer → SNS → SQS 1 → Consumer 1
              → SQS 2 → Consumer 2
              → SQS 3 → Consumer 3
  • SNS + SQS 組合
  • 可靠的訊息分發
  • 每個消費者獨立處理

4. Event-Driven（事件驅動）

Event Source → EventBridge → Lambda
                           → Step Functions
                           → SQS
  • 使用 EventBridge
  • 基於事件的鬆耦合架構
  • 支援複雜的事件路由

選擇適當的服務

需求推薦服務
簡單佇列SQS Standard
順序處理SQS FIFO
廣播通知SNS
事件路由EventBridge
複雜流程Step Functions
MQTT 協定Amazon MQ

解耦最佳實踐

使用佇列解耦

Web Tier → SQS → Processing Tier → SQS → Database Tier
  • 各層獨立擴展
  • 容錯能力提升
  • 尖峰流量緩衝

使用事件解耦

Order Service → SNS → Inventory Service
                    → Billing Service
                    → Notification Service
  • 服務獨立演進
  • 新增服務不影響現有服務

錯誤處理模式

Dead Letter Queue（DLQ）

  • 收集處理失敗的訊息
  • 支援 SQS、SNS、Lambda
  • 用於錯誤分析和重新處理

重試策略

策略說明
指數退避每次重試間隔增加
固定間隔固定時間重試
有限重試最大重試次數後放棄

流量控制

限流（Throttling）

  • API Gateway 限流
  • Lambda 並發限制
  • SQS 可見性超時

緩衝（Buffering）

  • SQS 作為緩衝層
  • Kinesis 處理串流資料

整合模式比較

模式優點缺點
同步 API簡單、即時耦合高、容錯低
非同步佇列解耦、容錯複雜度增加
事件驅動高度解耦除錯困難

最佳實踐

  1. 優先非同步：提高系統彈性
  2. 使用 DLQ：處理失敗訊息
  3. 實施冪等性：處理重複訊息
  4. 監控佇列深度：預防積壓
  5. 設計回壓機制：處理過載情況
  6. 記錄追蹤 ID：分散式追蹤

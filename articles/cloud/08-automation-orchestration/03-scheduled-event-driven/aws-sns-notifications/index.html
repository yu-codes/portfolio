概述

  • 發布/訂閱（PubSub）模型
  • Publisher 發送訊息到 SNS 主題
  • 所有 Subscriber 同步接收訊息
  • 支援最多 10 萬個 Topic
  • 每個 Topic 最多 1250 萬個訂閱者

訂閱端點類型

端點類型說明
Kinesis Firehose串流至資料湖
SQS佇列處理
HTTP/HTTPSWebhook
Lambda無伺服器處理
Email郵件通知
SMS簡訊通知
行動推播App 推播

訂閱過濾政策（Subscription Filter Policy）

  • 用 JSON 設定條件
  • 僅讓特定訊息送到特定訂閱者
  • 減少不必要的訊息傳遞

範例

{
  "eventType": ["order_placed"],
  "store": ["store-1", "store-2"]
}

Fan-out 模式

概念

SNS 主題對應多個 SQS 訂閱者，達成訊息分發。

架構

Producer → SNS Topic → SQS Queue 1 → Consumer 1
                    → SQS Queue 2 → Consumer 2
                    → SQS Queue 3 → Consumer 3

使用案例

訂單事件依狀態發送給三個服務處理：

  • 庫存服務
  • 帳單服務
  • 物流服務

優勢

  • 完全解耦生產者和消費者
  • 每個消費者獨立處理
  • 可擴展性高

SNS vs SQS 比較

項目SNSSQS
模式Pub/Sub佇列
訊息接收推送（Push）拉取（Poll）
持久化不持久（即時送達）持久化
消費者多個同時接收一個訊息一個消費者
適用場景廣播、通知任務處理、解耦

訊息屬性

屬性說明
MessageId訊息唯一識別碼
Subject訊息主題（Email 用）
MessageAttributes自訂屬性（用於過濾）
Timestamp發布時間

訊息大小

  • 最大 256 KB
  • 超過時可使用 Extended Client Library（儲存於 S3）

SNS 加密

  • 靜態加密：使用 SSE（服務端加密）
  • 傳輸中加密：使用 HTTPS 端點

FIFO Topic

  • 支援嚴格訊息順序
  • 訊息去重複
  • 訊息分組
  • 可與 FIFO SQS 佇列搭配使用

安全性

存取控制

  • IAM Policy：控制誰可以發布和訂閱
  • SNS Topic Policy：控制跨帳號存取

加密

  • 傳輸中加密：HTTPS
  • 靜態加密：使用 KMS

最佳實踐

  1. 使用過濾政策：減少不必要的訊息
  2. 搭配 SQS 實現 Fan-out：可靠的訊息分發
  3. 啟用加密：保護敏感訊息
  4. 監控指標：追蹤發布和傳遞狀態
  5. 使用 DLQ：處理傳遞失敗

SNS 架構深入解析

概述

  • 公共 AWS 服務
  • 高度可用、持久、安全、pub-sub 訊息服務

核心概念

概念說明
Topic訊息通道
Publisher發送訊息到 Topic
Subscriber接收 Topic 訊息
訊息大小最大 256 KB

Fan-out 架構詳解

問題場景

當一個事件需要觸發多個不同的處理流程：

訂單建立 → 需要通知：
           1. 庫存系統
           2. 帳單系統  
           3. 物流系統
           4. 客戶通知

SNS + SQS Fan-out

Producer → SNS Topic → SQS Queue 1 → Consumer 1（庫存）
                    → SQS Queue 2 → Consumer 2（帳單）
                    → SQS Queue 3 → Consumer 3（物流）
                    → Lambda → Email 通知

優勢

優勢說明
解耦Producer 不需知道 Consumer
可靠性SQS 提供訊息持久化
擴展性輕鬆新增 Subscriber
獨立處理每個 Consumer 獨立失敗/重試

訊息過濾詳解

概念

  • 每個 Subscriber 可設定過濾政策
  • 只接收符合條件的訊息
  • 減少不必要的處理

過濾政策範例

{
  "eventType": ["order_placed", "order_shipped"],
  "customer_tier": ["premium"]
}

支援的過濾條件

條件類型說明
精確匹配"key": ["value"]
前綴匹配"key": [{"prefix": "val"}]
數值範圍"key": [{"numeric": [">", 0, "<=", 100]}]
存在檢查"key": [{"exists": true}]

Kinesis 概述

什麼是 Kinesis

  • 可擴展的串流服務
  • 公共服務
  • 設計用於大規模即時資料處理

Kinesis 產品家族

產品用途
Kinesis Data Streams大規模資料擷取
Kinesis Data Firehose資料載入到目的地
Kinesis Data AnalyticsSQL 分析串流資料
Kinesis Video Streams影片串流處理

Kinesis Data Streams

架構

Producers → Kinesis Stream（Shards）→ Consumers

Shard 概念

特性說明
容量每 Shard 1 MB/s 輸入，2 MB/s 輸出
記錄每 Shard 1,000 records/s
擴展增加 Shard 數量

資料保留

  • 預設：24 小時
  • 最大：365 天

Producer 類型

  • Kinesis Producer Library（KPL）
  • AWS SDK
  • Kinesis Agent

Consumer 類型

  • Kinesis Client Library（KCL）
  • Lambda
  • Kinesis Data Analytics
  • Kinesis Data Firehose

Kinesis Data Firehose

概念

  • 完全託管的資料傳遞服務
  • 近即時（不是即時，有緩衝）
  • 自動擴展

支援的目的地

目的地說明
S3資料湖
Redshift資料倉儲（透過 S3）
OpenSearch搜尋和分析
HTTP 端點自訂目的地
Splunk日誌分析

Firehose vs Data Streams

項目FirehoseData Streams
管理完全託管需管理 Shard
延遲近即時（~60s）即時（~200ms）
資料保留不保留最多 365 天
用途資料載入自訂處理

Kinesis Data Analytics

功能

  • 使用 SQL 分析串流資料
  • 即時分析
  • 輸出到 Kinesis Streams 或 Firehose

使用案例

  • 即時儀表板
  • 異常偵測
  • 即時指標計算

概述

  • AWS 的事件匯流排服務
  • 前身為 CloudWatch Events
  • 用於建構事件驅動架構

核心概念

Event Bus（事件匯流排）

類型說明
DefaultAWS 服務事件的預設匯流排
Custom自訂應用程式事件
Partner第三方 SaaS 整合

事件來源

  • AWS 服務（EC2、S3、Lambda 等）
  • 自訂應用程式
  • SaaS 應用程式（Zendesk、Datadog 等）

規則（Rules）

  • 定義事件匹配模式
  • 指定目標處理

目標（Targets）

  • Lambda
  • SQS
  • SNS
  • Step Functions
  • Kinesis
  • API Gateway
  • 其他 AWS 服務

排程功能

Cron 表達式

  • 類似 Unix cron
  • 支援複雜的排程模式

Rate 表達式

  • 簡單的固定間隔
  • 例如：rate(5 minutes)

使用案例

  • 定時觸發 Lambda
  • 排程批次處理
  • 定期健康檢查

事件模式（Event Pattern）

範例

{
  "source": ["aws.ec2"],
  "detail-type": ["EC2 Instance State-change Notification"],
  "detail": {
    "state": ["stopped", "terminated"]
  }
}

Schema Registry

  • 自動發現事件結構
  • 產生程式碼綁定
  • 版本管理

與 CloudWatch Events 比較

項目CloudWatch EventsEventBridge
功能基本事件路由進階功能
Schema不支援支援
第三方整合有限豐富
建議舊專案新專案推薦

EventBridge 進階功能

EventBridge Pipes

  • 建立事件來源與目標之間的點對點整合
  • 支援可選的過濾和資料轉換
  • 無需編寫粘合程式碼

API Destinations

  • 將事件傳送到任何 HTTP API 端點
  • 支援 OAuth 和 API Key 認證
  • 內建重試和速率限制

EventBridge Archive

  • 長期儲存事件以供重播
  • 支援事件重播用於測試和復原
  • 可設定保留期限

AWS Step Functions

概述

  • 無伺服器工作流程編排服務
  • 使用 Amazon States Language（JSON）定義工作流程
  • 自動處理重試和錯誤

工作流程類型

類型說明適用場景
Standard Workflows長時間運行、持久、可稽核訂單處理、ETL
Express Workflows高吞吐量、短期執行串流處理、IoT

功能

  • 與超過 200 個 AWS 服務直接整合
  • Distributed Map：大規模並行資料處理
  • Workflow Studio：視覺化工作流程設計
  • 與 CloudWatch 和 X-Ray 整合監控

Amazon Kinesis

元件

服務說明
Kinesis Data Streams即時資料串流收集和處理
Kinesis Data Firehose將串流資料傳送到儲存目的地
Kinesis Data AnalyticsSQL 或 Apache Flink 處理串流資料
Kinesis Video Streams串流影片資料處理

Kinesis Data Streams 特性

  • 資料保留 1-365 天
  • 支援 KMS 加密
  • 可調整 Shard 數量擴展容量
  • 支援 On-demand 容量模式

Kinesis Data Firehose 目的地

  • Amazon S3
  • Amazon Redshift
  • Amazon OpenSearch Service
  • Splunk
  • 自訂 HTTP 端點

Amazon MQ

概述

  • AWS 託管的 Apache ActiveMQ 訊息代理服務
  • 支援 MQTT 協定
  • 協助既有系統遷移至 AWS

使用場景

  • 現有使用 ActiveMQ 的應用
  • 需要 MQTT 協定支援
  • 遷移傳統訊息佇列系統

vs SQS/SNS

項目Amazon MQSQS/SNS
協定JMS、AMQP、MQTT 等AWS 專有
管理需管理 Broker完全託管
適用遷移現有系統新建雲原生應用

最佳實踐

  1. 使用 EventBridge：新專案優先選擇
  2. 善用 Schema Registry：管理事件結構
  3. 實施事件過濾：減少不必要的處理
  4. 設定 DLQ：處理失敗事件
  5. 監控事件流：追蹤事件處理狀態
  6. 使用 EventBridge Pipes：簡化點對點整合
  7. 考慮 Step Functions：複雜工作流程編排

AWS Key Management Service (KMS)

  • 集中式金鑰管理服務，可產生、儲存與輪替金鑰（CMK）
  • 支援 對稱加密金鑰（加/解密用同一把）與 非對稱金鑰（公開/私密鑰）
  • 支援 自動金鑰輪替（每年一次）
  • KMS 每次呼叫最多加密 4KB 資料，更大的資料使用信封加密（Envelope Encryption）

KMS 金鑰類型

類型說明
AWS 管理金鑰AWS 自動建立與管理
客戶管理金鑰（CMK）客戶自行建立與管理
自訂金鑰存放區使用 CloudHSM 作為金鑰存放區
對稱金鑰AES-256 加密
非對稱金鑰RSA 或 ECC 金鑰對
HMAC 金鑰用於訊息驗證碼

使用場景

  • EBS 加密
  • S3 物件加密（SSE-KMS）
  • RDS 加密
  • Secrets Manager 加密
  • Lambda 環境變數加密

AWS CloudHSM

  • AWS 託管的 專用硬體安全模組 (Hardware Security Modules)
  • 自行控制加密金鑰（不可匯出）
  • 支援 PKCS#11 / JCE / CNG API
  • 可與 KMS 整合，自建 custom key store

CloudHSM vs KMS 比較

項目KMSCloudHSM
管理方式AWS 管理客戶管理
金鑰控制共享控制完全控制
合規性FIPS 140-2 Level 2FIPS 140-2 Level 3
整合原生 AWS 服務整合需自行整合
成本按使用量計費按小時計費

CloudHSM 使用場景

  • 需要完全控制金鑰的合規要求
  • FIPS 140-2 Level 3 合規需求
  • 需要高效能加密運算
  • 與 KMS 整合作為自訂金鑰存放區

加密最佳實踐

  1. 使用 KMS 進行大部分加密需求：簡單、成本效益高
  2. 敏感工作負載使用 CloudHSM：需要完全金鑰控制時
  3. 啟用金鑰輪替：定期更換金鑰降低風險
  4. 使用金鑰政策：控制誰可以使用哪些金鑰
  5. 監控金鑰使用：透過 CloudTrail 追蹤金鑰操作

方法說明
靜態加密（At Rest）資料儲存時的加密
傳輸中加密（In Transit）資料傳輸時的加密

類型說明範例
對稱加密同一把金鑰加解密AES-256
非對稱加密公鑰加密、私鑰解密RSA

KMS 架構深入解析

KMS 金鑰

類型說明
AWS Managed KeysAWS 建立、管理、使用
Customer Managed Keys客戶建立、客戶管理、可配置
AWS Owned KeysAWS 服務內部使用

KMS 金鑰組成

  • Key ID：唯一識別碼
  • 建立日期
  • Key Policy：資源政策
  • 描述和狀態

重要特性

  • KMS 金鑰永不離開 KMS
  • 僅對最大 4KB 資料直接加密
  • 金鑰受到 FIPS 140-2（L2） 保護

Data Encryption Keys（DEKs）

信封加密

步驟說明
1使用 GenerateDataKey 產生 DEK
2DEK 用於加密資料
3加密後的 DEK 與資料一起儲存
4解密時先解密 DEK，再用 DEK 解密資料

架構

KMS Key → 產生 DEK（明文 + 加密版本）
               ↓
           明文 DEK 加密資料 → 丟棄明文 DEK
               ↓
           儲存：加密的 DEK + 加密的資料

優點

  • 大資料量加密
  • KMS 只處理 DEK（小資料）
  • 減少 KMS API 呼叫

Key Policy 與授權

Key Policy（資源政策）

  • 每個 KMS 金鑰都有一個 Key Policy
  • 類似 S3 Bucket Policy
  • 預設允許帳號 Root 使用者完全存取

預設 Key Policy 範例

{
  "Sid": "Enable IAM User Permissions",
  "Effect": "Allow",
  "Principal": {"AWS": "arn:aws:iam::ACCOUNT-ID:root"},
  "Action": "kms:*",
  "Resource": "*"
}

授權模式

模式說明
Key Policy明確允許主體
Key Policy + IAM預設模式，兩者結合
Grants程式化授權，可撤銷

金鑰輪替

自動輪替

  • 僅適用於 Customer Managed Keys
  • AWS Managed Keys 每年自動輪替
  • 舊的金鑰保留用於解密
  • 新的加密使用新金鑰

手動輪替

  • 建立新的 KMS 金鑰
  • 更新應用程式參照
  • 使用別名簡化切換

多區域金鑰

概念

  • 相同的金鑰存在於多個區域
  • 一個主要金鑰 + 多個副本金鑰
  • 可在任一區域加解密

使用案例

  • 全域應用程式
  • 災難復原
  • 全域 DynamoDB Global Table 加密

限制

  • 非預設功能，需明確啟用
  • 有額外成本考量

AWS Secrets Manager

概述

  • 儲存和管理密鑰（密碼、API 金鑰等）
  • 支援自動輪替
  • 與 RDS、Redshift、DocumentDB 原生整合

與 SSM Parameter Store 比較

項目Secrets ManagerParameter Store
用途密鑰和憑證配置和密鑰
自動輪替原生支援需自行實作
RDS 整合是否
成本較高較低

架構

應用程式 → Secrets Manager API → 取得密鑰
                    ↓
               自動輪替 → 更新 RDS 密碼

CloudHSM 架構詳解

概述

  • 單租戶 HSM
  • AWS 管理硬體，客戶管理金鑰
  • 符合 FIPS 140-2 Level 3

架構

組件管理者
硬體維護AWS
軟體修補AWS
高可用AWS
金鑰管理客戶
使用者管理客戶

部署

  • 在 VPC 中運行
  • 透過 ENI 存取
  • 建議部署在多個 AZ

使用案例

場景說明
合規需求FIPS 140-2 Level 3
完全金鑰控制金鑰不與 AWS 共享
高效能需要專用加密硬體
自訂金鑰存放區與 KMS 整合

KMS 自訂金鑰存放區

  • CloudHSM 作為 KMS 的後端
  • 同時享有 KMS 整合和 HSM 安全性
  • 金鑰永不離開 HSM

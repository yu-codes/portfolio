概述

  • IAM 用來管理使用者與資源的存取權限
  • IAM 是全球性服務（全區域共用），免費使用
  • 預設建立的 Root 帳號 擁有完整權限，但不建議使用

核心元件

User（使用者）

  • 對應實際人員，應使用自己的帳號密碼登入 AWS 控制台

Group（群組）

  • 可包含多個使用者，不能巢狀其他群組

Role（角色）

  • 授權給受信任實體（如 EC2、Lambda、其他帳號、Web Identity、SAML 等）以暫時性存取 AWS 資源

Policy（政策）

  • JSON 文件，用來允許或拒絕特定使用者/群組/角色對 AWS 資源 的操作權限

Policy 結構

欄位說明
Version政策語言版本，目前為 2012-10-17
Statement政策內容的集合
Sid（可選）標記該條政策
EffectAllow 或 Deny
Principal指定授權對象（使用者、群組、角色）
Action允許執行的操作（如 s3:GetObject）
Resource指定作用的資源 ARN
Condition（可選）設定條件，例如 IAM 身份、來源 IP 等

Policy 範例

{
    "Version":"2012-10-17",
    "Statement":[{
        "Sid": "Deny-Barclay-S3-Access",
        "Effect":"Deny",
        "Principal": { "AWS": ["arn:aws:iam:123456789012:barclay"] },
        "Action": [ "s3:GetObject", "s3:PutObject", "s3:List*" ],
        "Resource": ["arn:aws:s3:::mybucket/*"]
    },{
        "Effect": "Allow",
        "Action": "iam:CreateServiceLinkedRole",
        "Resource": "*",
        "Condition": {
            "StringLike": {
                "iam:AWSServiceName": [
                    "rds.amazonaws.com",
                    "rds.application-autoscaling.amazonaws.com"
                ]
            }
        }
    }]
}

進階概念

Resource-based Policy（資源政策）

  • S3、SNS、SQS 可針對資源設政策

IAM Permission Boundary

  • 定義使用者/角色的最大權限邊界

IAM 評估順序

  1. 明確拒絕
  2. SCPs（Service Control Policies）
  3. 資源政策
  4. 權限邊界
  5. 使用者/角色政策

程式化存取 AWS

方式認證方式
AWS Management Console帳號密碼 + MFA
AWS CLI / SDKAccess Key ID + Secret Access Key
AWS CloudShell登入 AWS 使用（瀏覽器 CLI 工具）

AWS CLI 設定

$ aws configure
AWS Access Key ID [None]:
AWS Secret Access Key [None]:
Default region name [None]:
Default output format [None]:

非 IAM 使用者存取 AWS（Federation）

非 IAM 使用者需先經由聯邦身份驗證，透過 STS AssumeRole API 取得 IAM Role 所產生的臨時憑證（token）來存取 AWS。

聯邦身份驗證方式說明
SAML 2.0與 AD / ADFS 整合，使用 AssumeRoleWithSAML
自訂身份代理（Custom Identity Broker）當 IdP 不支援 SAML 時，使用 AssumeRole 或 GetFederationToken
Web Identity Federation使用 Amazon、Facebook、Google、OIDC IdP 登入後交換 token，使用 AssumeRoleWithWebIdentity
AWS CognitoAmazon 官方推薦身分驗證服務
AWS SSO單一登入服務，使用者可直接存取 AWS，不需呼叫 STS API

IAM 進階功能

IAM Roles Anywhere

  • 讓 AWS 外部的工作負載使用 IAM 角色
  • 適用於本地伺服器、容器等
  • 使用 X.509 憑證進行身份驗證

屬性式存取控制（ABAC）

  • 基於標籤屬性授予權限
  • 簡化大規模權限管理
  • 減少政策數量

IAM Access Analyzer

  • 識別與外部實體共享的資源
  • 分析 IAM 政策發現過度權限
  • 持續監控並產生發現

AWS Directory Service

可使用 AWS Directory Service 整合 Microsoft AD：

類型說明
AWS Managed Microsoft AD完整 AD 功能，適合需要與內部 AD 信任關聯的場景
AD Connector代理連線至內部 AD，不儲存資料
Simple AD基本 AD 功能，無法與內部 AD 連線

IAM Identity Policy 深入解析

Policy 結構與評估

  • 用於授予或拒絕使用者、群組、角色的存取權限
  • 當操作與資源匹配時，Statement 才會生效
  • 萬用字元（*）可匹配任何操作

優先順序

  1. 明確拒絕（Explicit DENY）：最高優先，永遠生效
  2. 明確允許（Explicit ALLOW）：次優先
  3. 預設拒絕（Default DENY）：未指定時預設拒絕
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "FullAccess",
            "Effect": "Allow",
            "Action": ["s3:*"],
            "Resource": ["*"]
        },
        {
            "Sid": "DenyCatBucket",
            "Effect": "Deny",
            "Action": ["s3:*"],
            "Resource": ["arn:aws:s3:::catgifs", "arn:aws:s3:::catgifs/*"]
        }
    ]
}

Inline Policy vs Managed Policy

類型說明適用場景
Inline Policy直接寫入使用者/群組/角色特殊例外情況
Managed Policy可重複使用，低管理開銷預設建議使用

IAM Users 與 ARN

身份認證流程

  1. Principal：嘗試存取 AWS 資源的實體
  2. 認證：透過帳密或 Access Key 驗證身份
  3. 已認證身份：通過認證後成為 Authenticated Identity
  4. 授權：IAM 檢查該身份是否有權執行操作

Amazon Resource Name（ARN）

ARN 用於唯一識別 AWS 資源：

arn:aws:partition:service:region:account-id:resource-id
arn:aws:partition:service:region:account-id:resource-type
arn:aws:partition:service:region:account-id:resource-type:resource-id

# 範例
arn:aws:s3:::catgifs          # Bucket 本身
arn:aws:s3:::catgifs/*        # Bucket 內的物件

⚠️ 這兩個 ARN 不重疊：前者是管理 bucket 的權限，後者是管理物件的權限

IAM Users 限制

  • 每個帳號最多 5000 個 IAM Users
  • 每個使用者最多加入 10 個群組
  • 大規模應用應使用 IAM Roles 與 Identity Federation

IAM Groups 詳解

  • Groups 是 使用者的容器，便於管理
  • 群組可附加 Identity Policy
  • 沒有「所有使用者」的原生群組（但可自行建立）
  • 限制：最多 300 個群組，每使用者最多 10 個群組
  • 不支援巢狀群組
  • Resource Policy 無法直接授權給群組

IAM Roles 深入解析

角色特性

  • 適用於未知數量的 Principal 或超過 5000 使用者
  • 角色是被 假設（Assume） 的身份
  • 有兩種 Policy：
    • Trust Policy：定義誰可以假設此角色
    • Permissions Policy：定義角色擁有的權限

STS（Security Token Service）

  • 產生臨時安全憑證
  • 使用 sts:AssumeRole API
  • 憑證過期後需重新假設角色

角色使用場景

場景說明
AWS 服務Lambda Execution Role、EC2 Instance Role 等
緊急存取Break Glass 場景，平時只讀權限的團隊可臨時獲取寫入權限
大型組織超過 5000 身份時使用 Identity Federation
Web 應用Web Identity Federation，使用社交帳號登入
跨帳號存取Cross-Account Access

Service-linked Roles 與 PassRole

Service-linked Roles

  • 直接連結到特定 AWS 服務的 IAM 角色
  • 由服務預先定義，包含所需的所有權限
  • 無法在使用中時刪除

PassRole

  • 授權使用者將角色傳遞給 AWS 服務
  • 防止使用者透過服務獲取超出自身的權限
  • 重要的安全控制機制

AWS Organizations

架構概念

  • 使用標準 AWS 帳號建立組織
  • 該帳號成為 管理帳號（Management Account），唯一一個
  • 組織不在 AWS 帳號「內部」
  • 可邀請或直接建立成員帳號
  • Organization Root 是容器，包含帳號或組織單位（OU）

主要功能

  • 統一帳單（Consolidated Billing）：成員帳號的費用由管理帳號支付
  • 移除財務管理開銷
  • 整合預留和批量折扣
  • 可直接在組織內建立帳號
  • 使用 IAM Roles 跨帳號切換，無需為每個帳號建立 IAM Users

Service Control Policies（SCP）

SCP 概念

  • JSON 格式的政策文件
  • 可附加到整個組織、OU 或帳號
  • 向下繼承到所有下層 OU 和帳號
  • 管理帳號不受 SCP 影響

SCP 特性

  • SCP 是 帳號權限邊界
  • 限制帳號（包括 root user）可以執行的操作
  • SCP 不授予任何權限，只限制權限

允許清單 vs 拒絕清單

架構說明
拒絕清單（預設）FullAWSAccess 預設開啟，明確拒絕特定服務
允許清單移除 FullAWSAccess，只允許特定服務（管理開銷較大）

最佳實踐：使用拒絕清單架構

AWS Control Tower

概述

  • 快速設定多帳號環境
  • 整合 Organizations、IAM Identity Center、CloudFormation、Config 等服務
  • Landing Zone：多帳號環境
  • Guard Rails：偵測/強制規則與標準
  • Account Factory：自動化帳號建立
  • Dashboard：環境總覽

Landing Zone

  • Well-Architected 多帳號環境
  • Security OU：Log Archive 和 Audit 帳號
  • Sandbox OU：測試環境，安全限制較寬鬆
  • IAM Identity Center：SSO、多帳號、Identity Federation

Guard Rails

  • 強制（Mandatory）、強烈建議（Strongly Recommended） 或 選用（Elective）
  • 預防性（Preventive）：透過 SCP 阻止操作
  • 偵測性（Detective）：透過 Config Rules 進行合規檢查

Account Factory

  • 自動化帳號佈建
  • 自動套用 Guard Rails
  • 可與 SDLC（軟體開發生命週期）整合 | Amazon Cognito | 用於 App 使用者註冊/登入，不是 Microsoft AD |

Amazon Cognito

Cognito User Pool（用戶池）

  • 提供 App 使用者的身份驗證功能，回傳 JWT token
  • 支援社群登入（Google/Facebook/Apple）與 OIDC/SAML
  • JWT 可用於：
    • 換取 AWS 臨時憑證（存取資源）
    • 存取 API Gateway（搭配 IAM Role）

Cognito Identity Pool（身份池）

  • 用來授權 AWS 資源使用權限
  • 支援 guest user、透過 federated identity 提供 S3/其他資源臨時權限

SSL/TLS 憑證管理

  • 若使用第三方 CA 的 SSL/TLS 憑證，應匯入到 AWS Certificate Manager (ACM) 或 IAM 憑證儲存區

最佳實踐

  1. 遵循最小權限原則（Least Privilege）：只給需要的權限
  2. 啟用 MFA：為 Root 帳號和重要使用者啟用多因素認證
  3. 定期輪替 Access Key：避免憑證外洩風險
  4. 使用 Role 而非 User：EC2、Lambda 等服務應使用 Role
  5. 使用 IAM Access Analyzer：持續監控資源存取
  6. 考慮 ABAC：簡化大規模權限管理
  7. 使用 IAM Roles Anywhere：本地工作負載存取 AWS

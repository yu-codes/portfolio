RDS Multi-AZ

運作方式

  • 在不同 AZ 建立同步備援實例
  • 發生故障時自動將 CNAME 指向備援實例
  • 故障轉移時間通常為 1-2 分鐘

特性

特性說明
複製方式同步
備援用途僅用於故障轉移，不處理讀取
費用需額外費用
適用引擎所有 RDS 引擎

Aurora 高可用性

Aurora 儲存架構

  • 資料自動複製到 3 個 AZ
  • 每個 AZ 2 份副本，共 6 份
  • 可容忍最多 2 份副本遺失（寫入）
  • 可容忍最多 3 份副本遺失（讀取）

Aurora 故障轉移

  • 有 Replica 時：提升 Replica 為主節點（通常 <30 秒）
  • 無 Replica 時：建立新主節點（較慢）

備份策略

RDS 自動備份

項目說明
頻率每日
保留期1-35 天
Console 預設7 天
CLI 預設1 天
類型增量備份

RDS 手動快照

  • 手動建立，不自動過期
  • 可跨區域複製
  • 可分享給其他帳號

Aurora 備份

  • 自動連續備份至 S3
  • 保留期 1-35 天
  • 支援時間點還原（PITR）

AWS Backup

集中管理並自動備份下列資源：

  • EC2
  • EBS
  • EFS
  • RDS
  • DynamoDB
  • FSx（Lustre/Windows）
  • Storage Gateway

使用案例

  • 備份 RDS 並保留 90 天（RDS 原生自動備份最多僅支援 35 天）
  • 跨區域備份
  • 集中管理多帳號備份

災難復原策略

策略類型

策略RTORPO成本
Backup & Restore小時級最後備份最低
Pilot Light十分鐘級接近即時低
Warm Standby分鐘級即時中
Multi-Site Active即時即時最高

Aurora Global Database

  • RPO：1 秒
  • RTO：1 分鐘
  • 適用於跨區域災難復原

串流與分析服務

Amazon Kinesis

全託管即時資料串流服務。

類型說明
Kinesis Data Streams接收與儲存即時資料流，保留期 24 小時～365 天
Kinesis Data Firehose直接導入 S3、Redshift、ElasticSearch，可搭配 Lambda 轉換
Kinesis Data Analytics使用 SQL 或 Apache Flink 分析資料流
Kinesis Video Streams接收與處理即時影音串流

Kinesis Data Streams

  • 生產者：Kinesis Agent、SDK、KPL
  • 消費者：Kinesis Analytics、Firehose、KCL

Amazon Redshift

  • 以欄式儲存為主的 OLAP 分析型資料庫
  • 支援 MPP（Massive Parallel Processing）平行查詢
  • 適合大數據分析與資料倉儲
  • 能整合 AWS Quicksight、Tableau 等 BI 工具
  • Redshift Spectrum：直接查詢 S3 上的資料

Amazon EMR

  • 全託管大數據平台
  • 支援 Hadoop、Spark、Hive、Flink、Presto 等開源工具
  • 適合資料轉換（ETL）與分析

最佳實踐

  1. 生產環境使用 Multi-AZ：確保高可用性
  2. 配置適當的備份保留期：符合合規要求
  3. 定期測試還原流程：確保備份可用
  4. 使用 AWS Backup：集中管理備份
  5. 考慮跨區域備份：災難復原需求
  6. Aurora Global Database：需要最低 RPO/RTO 時

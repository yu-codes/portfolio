訓練一個好模型只是成功的一半。如何讓它高效、穩定地服務生產流量，是同等重要的挑戰。

推論模式

即時推論（Real-time / Online）

用戶發送請求，立即獲得預測結果。

典型場景：推薦系統、風控系統、聊天機器人

要求：

  • 低延遲（通常 < 100ms）
  • 高可用性
  • 能處理突發流量

批次推論（Batch / Offline）

定期對大量資料進行預測，結果儲存供後續使用。

典型場景：每日信用評分更新、每週用戶分群

要求：

  • 高吞吐量
  • 成本效率
  • 可排程

近即時推論（Near Real-time）

在事件發生後幾秒到幾分鐘內處理。

典型場景：異常偵測、串流資料處理

如何選擇？

考量即時批次
延遲要求毫秒級無限制
吞吐量中極高
成本常駐服務，成本高按需啟動，成本低
結果新鮮度最新有延遲

即時推論架構

簡單架構

REST API 伺服器 + 模型。適合流量不大的場景。

負載均衡架構

多個推論伺服器 + 負載均衡器。支援水平擴展。

微服務架構

每個模型是一個獨立的微服務。適合有多個模型需要組合的場景。

好處：各模型獨立部署、獨立擴展、獨立更新。

GPU 推論架構

GPU 的特性是批次處理效率高。單一請求送到 GPU 是浪費——最好收集多個請求，一次送入 GPU。

動態批次（Dynamic Batching）：在短時間窗口內收集請求，湊成一批送進 GPU。Triton Inference Server 原生支援。

批次推論架構

Spark / MapReduce

用大資料處理框架把模型應用到海量資料上。

AWS Batch / SageMaker Batch Transform

雲端的託管批次推論服務。按用量計費。

分區處理

把資料按日期、用戶群等維度分區，平行處理各分區。

模型組合模式

串聯（Pipeline / Sequential）

模型 A 的輸出作為模型 B 的輸入。

例如：OCR → 文字分類 → 情感分析

並聯（Ensemble）

多個模型同時預測，結果做加權平均或投票。

A/B 測試

把流量分給不同模型，比較業務效果。

特徵計算分離

預測時需要的特徵可能來自多個來源（資料庫、外部 API、即時計算）。把特徵計算與模型推論分離，能提升系統的彈性。

效能優化

模型層面

  • 量化、剪枝（見模型打包章節）
  • 使用更高效的模型架構

推論引擎

  • ONNX Runtime：跨平台高效推論
  • TensorRT：NVIDIA GPU 專用，極致效能
  • OpenVINO：Intel 硬體專用

系統層面

  • 連接池（Connection Pooling）
  • 非同步處理
  • 快取（相同輸入直接返回快取結果）
  • 預計算（把不常變動的特徵預先算好）

實務心法

  1. 先問「需要即時還是批次？」——大多數場景批次就夠了，而且便宜很多
  2. GPU 推論一定要用動態批次——否則 GPU 利用率可能不到 10%
  3. 監控推論延遲的 P99，不是平均值——尾部延遲才是用戶體驗的殺手
  4. 快取是最便宜的效能提升——如果相同輸入很常見的話

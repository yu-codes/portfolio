在即時推論場景中，延遲就是一切。用戶不會等待一個思考 2 秒的推薦系統。每降低 10ms 延遲，都可能帶來可衡量的業務價值。

延遲預算分配

一次完整的預測請求通常包含：

  1. 網路傳輸（10-50ms）
  2. 特徵擷取（5-50ms）
  3. 特徵預處理（1-10ms）
  4. 模型推論（5-500ms）
  5. 後處理（1-5ms）

要先搞清楚延遲的瓶頸在哪，再對症下藥。不要還沒分析就急著優化模型——也許瓶頸是資料庫查詢。

模型級優化

量化

把 float32 降為 int8，推論速度通常提升 2-4 倍，模型大小縮小 4 倍。

動態量化：最簡單，只量化權重，啟動值運行時計算。 靜態量化：需要校準資料集，量化權重和啟動值。速度更快。 量化感知訓練：在訓練時模擬量化效果。精度保持最好。

知識蒸餾

把大模型的知識轉移到小模型。小模型推論更快，且效能接近大模型。

著名案例：DistilBERT 比 BERT 小 60%、快 60%，保留 97% 的效能。

架構優化

  • 使用更高效的架構（MobileNet vs ResNet）
  • 減少不必要的層
  • 使用深度可分離卷積等輕量級操作

TensorRT 的優化包括：

  • 層融合（把多個操作合併成一個核心）
  • 核心自動調整（auto-tuning）
  • 精度校準（FP16/INT8）

系統級優化

非同步處理

特徵計算和模型推論可以平行進行。用 async/await 或多執行緒，讓等待 I/O 的時間不阻塞 CPU。

快取

對相同或相似的輸入，直接返回快取的結果。

策略：

  • 精確快取：完全相同的輸入
  • 近似快取：把輸入離散化後快取
  • 特徵快取：把計算成本高的特徵預先快取

預計算

把不常變動的部分預先計算好。

例如：推薦系統可以預先計算用戶向量，請求時只需要做向量相似度搜尋。

模型預熱

模型第一次推論通常比較慢（記憶體分配、JIT 編譯等）。在接受流量前先用假資料跑幾次。

連接復用

保持長連接，避免每次請求都重新建立連接。

批次推論優化

動態批次

等待一小段時間，收集多個請求一起處理。GPU 的並行能力讓批次推論幾乎不增加單位延遲。

需要平衡：等待時間太長增加延遲，太短批次太小浪費 GPU。

非同步批次

不讓用戶等待批次湊滿，而是立即返回一個 token，用戶稍後用 token 來取結果。

延遲監控

關鍵指標

  • P50（中位數）：一般表現
  • P95：大多數用戶的體驗
  • P99：尾部延遲，通常是問題的訊號
  • P99.9：極端情況

P99 vs 平均值

永遠關注 P99 而非平均值。平均延遲 10ms 聽起來很好，但如果 P99 是 500ms，代表 1% 的用戶等了半秒以上。

實務心法

  1. 先找到瓶頸——不要盲目優化
  2. 量化是 ROI 最高的優化——幾乎免費的效能提升
  3. TensorRT 在 NVIDIA GPU 上是必用的——通常帶來 2-5 倍加速
  4. P99 延遲比平均延遲重要得多——用戶記住的是最差體驗

程式碼有 Git，但資料呢？資料版本控制長期被忽略，卻是確保 ML 實驗可重現性的關鍵。當你發現模型效能下降時，能不能回溯到「之前那個版本的資料」？

為什麼需要資料版本控制？

可重現性

ML 的結果取決於三個要素：程式碼、資料、環境。Git 管理了程式碼，Docker 管理了環境，但資料的版本往往是缺失的一環。

沒有資料版本控制，你可能遇到：

  • 「我明明用同樣的程式跑，結果不一樣」——因為資料在這段時間被更新了
  • 「上個月的模型比較好，但我找不到當時的訓練資料了」

稽核與合規

在金融、醫療等受監管行業，你需要能證明「這個模型是用什麼資料訓練的」。資料版本控制提供了這個證據。

資料版本控制工具

DVC（Data Version Control）

最廣泛使用的資料版本控制工具，設計理念是成為「資料的 Git」。

核心概念：

  • 資料檔案不直接存入 Git，而是存入遠端儲存（S3、GCS 等）
  • Git 中只存放一個小型的 .dvc 檔案，包含資料的雜湊值
  • 切換 Git 分支時，DVC 會自動切換對應版本的資料

Delta Lake

Databricks 推出的開源資料湖格式。在 Parquet 之上加入了 ACID 交易支援和版本控制。

特色：

  • 時間旅行（Time Travel）：查詢任意歷史版本的資料
  • ACID 交易：確保寫入的原子性
  • Schema Evolution：安全地變更資料結構

其他方案

工具特色

資料血緣（Data Lineage）

資料血緣回答一個關鍵問題：「這個資料是怎麼來的？」

為什麼重要？

  • 除錯：模型效能下降，是資料品質問題嗎？問題出在哪一步？
  • 影響分析：如果改了上游的資料源，哪些下游任務會受影響？
  • 信任：使用者需要知道資料經過了哪些處理，才能信任分析結果

追蹤什麼？

  • 技術血緣：表 A 經過什麼 SQL 轉換變成表 B
  • 業務血緣：某個指標的計算邏輯是什麼
  • 欄位級血緣：下游的某個欄位，來自上游的哪個欄位

工具選擇

  • OpenLineage：開放標準，與 Airflow、Spark 整合
  • Apache Atlas：Hadoop 生態系的元資料管理
  • dbt：在 SQL 轉換層自動追蹤血緣
  • DataHub：LinkedIn 開源的資料目錄

元資料管理（Metadata Management）

元資料是「關於資料的資料」，包括：

  • 結構性元資料：欄位名稱、類型、約束
  • 操作性元資料：建立時間、更新頻率、資料量
  • 業務性元資料：欄位含義、資料擁有者、使用政策

好的元資料管理讓團隊能快速找到需要的資料、理解它的含義和品質。

實務心法

  1. 把資料版本控制當作專案的第一天就要做的事，不要等到出問題才想起來
  2. 即使只有你一個人，也要記錄每次實驗使用的資料版本
  3. 資料血緣不需要一步到位，先從最關鍵的管線開始追蹤
  4. 自動化勝過手動文件——用工具自動追蹤，而非依賴人工記錄

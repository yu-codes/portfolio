自動化訓練

從手動實驗到自動化管線：建構可重複、可擴展的訓練流程

手動訓練模型在原型階段是可以的，但一旦進入生產，你需要自動化的訓練管線。因為模型需要定期重新訓練，而且你不想每次都手動操作。

為什麼要自動化訓練？

模型衰退（Model Decay）

模型上線後，效能會隨時間下降。原因：

  • 資料漂移（Data Drift）：輸入資料的分佈改變了。例如用戶行為隨季節變化。
  • 概念漂移（Concept Drift）：輸入和輸出的關係改變了。例如疫情改變了消費模式。
  • 上游系統變化：依賴的資料源格式或含義改變。

自動化重新訓練是應對模型衰退的核心策略。

重新訓練的觸發方式

觸發方式說明適用場景
定時觸發每週/每月自動重訓穩定的環境
效能觸發模型效能下降到閾值時動態環境
資料觸發新資料量達到門檻時資料驅動場景
漂移觸發偵測到資料漂移時需要即時反應

自動化訓練管線的架構

一個完整的自動化訓練管線包含：

資料準備階段

  • 從資料源擷取最新資料
  • 資料品質檢查
  • 特徵工程
  • 資料分割（train/validation/test）

訓練階段

  • 載入超參數配置
  • 執行模型訓練
  • 記錄實驗指標
  • 儲存模型產物

評估階段

  • 在測試集上評估
  • 與當前生產模型比較
  • 執行公平性和偏差檢查
  • 生成評估報告

部署階段

  • 如果新模型更好，推送到模型註冊中心
  • 執行 A/B 測試或漸進式部署
  • 更新監控設定

管線工具

Kubeflow Pipelines

在 Kubernetes 上運行 ML 管線。適合大規模、多團隊的場景。

MLflow Pipelines

輕量級的管線框架。與 MLflow 生態系統整合。

Airflow + ML 任務

用通用的工作流程引擎來編排 ML 訓練。彈性最大。

SageMaker Pipelines / Vertex AI Pipelines

雲端原生的 ML 管線服務。如果你已經在使用這些雲端平台，是自然的選擇。

持續訓練（Continuous Training）

類比於軟體工程的持續整合/持續部署（CI/CD），ML 有持續訓練（CT）。

CT 的核心是：新資料進來 → 自動觸發訓練 → 自動評估 → 自動部署（如果通過）。

CT 的挑戰

  • 計算成本：訓練不像編譯那麼便宜
  • 評估可靠性：自動化評估必須足夠全面
  • 安全閘門：某些情況下需要人工審核
  • 資料品質：壞資料可能訓練出壞模型

特徵儲存（Feature Store）

特徵儲存是連接資料工程和模型訓練的橋樑。

它解決的問題：

  • 訓練時和推論時使用的特徵計算邏輯不一致
  • 重複計算相同特徵
  • 無法回溯某個時間點的特徵值

主流工具：Feast（開源）、Tecton、Hopsworks、SageMaker Feature Store。

實務心法

  1. 先手動跑通整個流程，再自動化——不要自動化一個你不完全理解的流程
  2. 自動化訓練必須有人工審核的閘門——至少在初期
  3. 監控比自動化更重要——沒有監控的自動化是災難
  4. 投資在特徵儲存上——它能避免訓練-推論偏差

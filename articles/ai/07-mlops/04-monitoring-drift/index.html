監控與漂移偵測

模型上線後的守護：效能監控、資料漂移偵測與告警策略

模型上線不是終點，而是新旅程的起點。生產環境中的模型需要持續監控，因為它的效能會隨時間衰退。

模型衰退的原因

資料漂移（Data Drift）

輸入資料的分佈改變了，但輸入和輸出的關係沒變。

例如：你的推薦模型訓練時用的是 25-35 歲的用戶資料，但現在 18-24 歲的用戶快速增長。特徵分佈改變了。

概念漂移（Concept Drift）

輸入和輸出的關係改變了。

例如：疫情前「經常旅行」代表高消費力，疫情期間這個關係可能反轉。

特徵漂移

上游系統變化導致特徵值異常。例如某個 API 的回傳格式改了，某個欄位變成 null。

監控什麼？

模型效能指標

如果有真實標籤（ground truth），直接監控：

  • 準確率/AUC 的變化趨勢
  • 預測分佈的變化
  • 錯誤案例的分析

問題：真實標籤通常有延遲（幾天到幾個月）。你不能等到知道模型已經壞了才行動。

輸入資料分佈

即時監控輸入資料的統計特性：

  • 各特徵的均值、中位數、標準差
  • 類別特徵的頻率分佈
  • 缺失率
  • 極端值比例

預測分佈

監控模型輸出的分佈。如果預測分佈突然改變（例如陽性率從 5% 變成 20%），即使你不知道真實答案，也知道有問題。

系統指標

  • 推論延遲（P50, P95, P99）
  • 請求量
  • 錯誤率
  • GPU/CPU/記憶體使用率

漂移偵測方法

統計檢定

  • KS 檢定（Kolmogorov-Smirnov）：比較兩個分佈是否相同。適合連續特徵。
  • 卡方檢定：適合類別特徵。
  • PSI（Population Stability Index）：業界常用的穩定性指標。PSI > 0.2 通常代表顯著漂移。

距離度量

  • KL 散度：衡量兩個分佈的差異。不對稱，方向不同結果不同。
  • JS 散度：KL 散度的對稱版本。
  • Wasserstein 距離：「搬土距離」，有直覺的幾何解釋。

多維度漂移偵測

單獨看每個特徵可能遺漏多維度的漂移。需要同時考慮所有特徵的聯合分佈。

方法：

  • Maximum Mean Discrepancy（MMD）
  • 基於自編碼器的異常偵測
  • 資料密度估計

告警策略

分級告警

級別條件動作
資訊輕微偏移記錄，人工觀察
警告中度偏移通知團隊，開始調查
嚴重重大偏移或效能驟降自動回滾或降級

避免告警疲勞

  • 設定合理的閾值——太敏感會產生大量假警報
  • 用移動窗口平滑指標——避免偶發波動觸發告警
  • 分組告警——把相關的告警合併
  • 建立 on-call 輪班——確保告警有人處理

實務心法

  1. 沒有監控的模型不該上生產——這是底線
  2. 預測分佈是最即時的警報信號——不需要等真實標籤
  3. 定期回顧監控閾值——隨著系統成熟，閾值需要調整
  4. 自動化降級策略——當模型不可靠時，用規則型系統或上一版模型兜底

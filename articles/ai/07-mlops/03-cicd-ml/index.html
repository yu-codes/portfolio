ML 的 CI/CD

把軟體工程的 CI/CD 實踐應用到機器學習：測試、驗證與自動部署

軟體工程已經有成熟的 CI/CD 文化，但 ML 的 CI/CD 更複雜——因為你不只是在部署程式碼，還在部署資料和模型。

ML CI/CD 的特殊挑戰

傳統軟體的 CI/CD：程式碼變更 → 測試 → 建構 → 部署

ML 的 CI/CD 多了幾個維度：

  • 程式碼變更可能影響模型效能
  • 資料變更也會觸發重新部署
  • 模型的「測試」不只是單元測試，還包括效能評估
  • 部署需要考慮模型大小和推論延遲

ML 的測試金字塔

單元測試

測試程式碼邏輯的正確性：

  • 資料前處理函數
  • 特徵工程邏輯
  • 自定義損失函數
  • 自定義指標計算

資料測試

驗證資料的品質和一致性：

  • Schema 驗證：欄位是否存在、類型是否正確
  • 統計驗證：分佈是否在預期範圍內
  • 完整性驗證：缺失率是否在閾值內

工具：Great Expectations、Pandera。

模型測試

不只是看準確率，還要驗證：

  • 效能基準：至少不比當前生產模型差
  • 公平性：在不同群體上的表現是否一致
  • 穩健性：對微小輸入擾動是否敏感
  • 推論速度：延遲是否在 SLA 範圍內
  • 模型大小：是否能在目標環境部署

整合測試

端到端驗證：

  • 輸入一筆真實資料
  • 經過完整的預處理管線
  • 得到模型預測
  • 後處理輸出
  • 驗證整個流程的正確性

CI 管線設計

典型的 ML CI 管線：

  1. 程式碼品質檢查：linting、型別檢查
  2. 單元測試：所有程式碼邏輯
  3. 資料驗證：抽樣檢查資料品質
  4. 快速訓練測試：用小資料集快速訓練，驗證管線能跑通
  5. 模型效能測試：在基準資料集上評估

要點

  • CI 要快——不要在 CI 中做完整訓練（太慢）
  • 用小資料集或模擬資料做 smoke test
  • 效能測試可以在 merge 後的夜間建構中做

CD 管線設計

漸進式部署

不要一次切換所有流量到新模型：

  1. Shadow Mode：新模型在背景運行，不影響用戶，比較結果
  2. Canary 部署：把 1-5% 的流量導向新模型
  3. 漸進式擴大：效能確認後，逐步增加流量
  4. 全量切換：100% 流量到新模型
  5. 保留回滾能力：如果出問題，能快速切回舊模型

自動回滾

監控部署後的關鍵指標（延遲、錯誤率、業務指標）。如果異常，自動回滾到上一個版本。

實務心法

  1. ML 的 CI/CD 不是選配，是必配——沒有它，你的模型更新永遠是痛苦的
  2. 資料測試和模型測試同等重要——壞資料進去，壞模型出來
  3. Shadow Mode 是最安全的部署方式——先觀察再切換
  4. 永遠保留快速回滾的能力——不是所有問題都能在上線前發現

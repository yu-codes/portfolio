模型上線不是終點，而是長期營運的開始。如何持續維護和更新模型，決定了它能為業務帶來多長時間的價值。

模型更新策略

定期重訓

最簡單的策略：每隔固定時間（每週/每月）用最新資料重新訓練。

適合：環境變化穩定的場景 優點：簡單可預測 缺點：可能對突發變化反應不及時

觸發式更新

當偵測到特定條件時觸發重訓：

  • 模型效能低於閾值
  • 資料漂移超過門檻
  • 新資料量達到一定規模
  • 重大業務事件（如疫情、大促銷）

持續學習（Online Learning）

模型持續從新資料中學習，不斷更新。

適合：快速變化的環境（如新聞推薦、廣告排序） 挑戰：穩定性、災難性遺忘、資料品質

增量訓練 vs 完全重訓

增量訓練：在已有模型的基礎上，用新資料繼續訓練。快但可能累積偏差。

完全重訓：用全部歷史資料從頭訓練。慢但結果最可靠。

建議：定期做完全重訓，中間用增量訓練填補。

生命週期管理

模型的生命階段

  1. 開發階段：實驗、迭代、驗證
  2. 部署階段：A/B 測試、漸進式發布
  3. 穩定運行階段：持續監控、定期更新
  4. 衰退階段：效能下降、需要升級或替換
  5. 退役階段：停止服務、保留記錄

退役策略

什麼時候該退役一個模型？

  • 有更好的替代模型
  • 業務需求改變，模型不再適用
  • 模型的維護成本超過了它的價值
  • 法規或合規要求

退役時要確保：

  • 平滑遷移到新方案
  • 保留模型記錄和產物（供稽核和回溯）
  • 通知所有使用者

營運團隊的角色

ML 平台團隊

提供基礎設施和工具：

  • 訓練平台
  • 模型服務平台
  • 監控和告警系統
  • CI/CD 管線

ML 工程師

維護具體的模型：

  • 開發和更新模型
  • 分析模型效能
  • 處理告警和問題

值班制度

ML 模型需要 on-call 支援：

  • 誰負責處理模型相關的告警？
  • 告警的升級路徑是什麼？
  • 回滾的決策流程是什麼？

知識管理

實驗日誌

記錄每次模型改動的：

  • 動機和假設
  • 方法和實作細節
  • 結果和分析
  • 決策和下一步

模型決策文件

重要的模型決策應該有書面記錄：

  • 為什麼選擇這個架構？
  • 為什麼使用這些特徵？
  • 已知的限制是什麼？
  • 未來的改進方向？

事後回顧（Postmortem）

模型出了問題後：

  • 發生了什麼？
  • 時間線
  • 根本原因
  • 改善措施
  • 經驗教訓

成本管理

訓練成本

  • 使用 Spot 實例
  • 避免不必要的實驗
  • 優化超參數搜尋效率

推論成本

  • 選擇適當的實例大小
  • 自動擴縮
  • 模型量化降低計算需求
  • 快取減少重複計算

ROI 追蹤

定期評估每個模型的投入產出比：

  • 這個模型帶來了多少業務價值？
  • 維護成本是多少？
  • 是否值得繼續投資？

實務心法

  1. 模型營運比模型開發更耗時間——做好心理準備
  2. 自動化是減輕營運負擔的關鍵——手動操作遲早會出錯
  3. 知識管理不是浪費時間——未來的你會感謝現在的記錄
  4. 退役模型也是一種進步——不要讓過時的模型佔據資源

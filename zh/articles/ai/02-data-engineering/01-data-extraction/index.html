資料擷取

從各種來源獲取資料的策略與實務：API、資料庫、檔案、網頁爬蟲

資料是 ML 的燃料。但在你能訓練模型之前，首先得把資料從各種來源「搬」出來。資料擷取是整個資料管線的第一步，也是最容易被忽略的一步。

資料來源的多樣性

現實世界的資料散佈在各種地方：

批次擷取 vs 串流擷取

批次擷取（Batch）

定期（每小時、每天）一次性搬運大量資料。適合對即時性要求不高的場景。

常見模式：

  • 全量匯出：每次匯出整個表。簡單但浪費資源
  • 增量匯出：只匯出上次以來的變更。需要有時間戳或版本欄位
  • 快照比較：比較兩次快照的差異

串流擷取（Streaming）

資料一產生就即時傳輸。適合需要即時反應的場景。

  • CDC（Change Data Capture）：監聽資料庫的變更日誌，即時捕捉新增、修改、刪除事件。Debezium 是最常用的開源工具。
  • 事件串流：從 Kafka、Kinesis 等平台消費事件。

如何選擇？

考量批次串流
延遲要求分鐘到小時秒級
實作複雜度低高
資源成本尖峰集中平均分散
錯誤處理重跑整批需要處理亂序和重複

API 擷取的實務考量

從外部 API 擷取資料是最常見的場景之一，但有許多陷阱：

速率限制（Rate Limiting）：大多數 API 有請求頻率限制。你需要實作退避策略（exponential backoff），避免被封鎖。

分頁處理：大量資料通常需要分頁取得。注意：有些 API 在你分頁期間如果有新資料插入，可能會造成重複或遺漏。

認證與授權：OAuth、API Key、JWT——不同的 API 有不同的認證方式。要安全地管理密鑰，絕對不要硬編碼在程式中。

冪等性與重試：網路請求可能失敗。設計你的擷取程式時，要確保重試不會造成重複資料。

資料庫擷取策略

直接查詢：最簡單的方式，但對生產資料庫有壓力。建議在離峰時段執行，或使用讀取副本。

匯出與匯入：先把資料匯出成檔案（CSV、Parquet），再載入到分析環境。適合大量資料。

CDC：最優雅的方式。不影響生產資料庫效能，能即時捕捉變更。但需要額外的基礎設施。

檔案格式的選擇

格式特性適用場景

重點：如果資料量超過幾 GB，強烈建議使用 Parquet 而非 CSV。壓縮率可以差 5-10 倍，查詢速度也快很多。

實務心法

  1. 永遠要設計冪等的擷取程式——重跑不會產生重複資料
  2. 記錄擷取的 metadata：時間、資料量、來源版本，方便除錯和稽核
  3. 區分「擷取」和「轉換」——在 ELT 架構中，先原封不動地搬資料，轉換留到後面做
  4. 監控擷取程式：資料量異常（突然變成零或暴增）通常代表上游有問題

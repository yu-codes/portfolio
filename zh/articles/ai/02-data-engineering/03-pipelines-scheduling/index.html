資料管線（Data Pipeline）是把資料從 A 點搬到 B 點、同時進行必要轉換的自動化流程。設計良好的管線是資料工程的核心，直接影響下游模型和分析的品質與時效。

ETL vs ELT：兩種架構思維

ETL（Extract-Transform-Load）

先擷取、再轉換、最後載入。傳統資料倉儲的做法。

資料在進入目標系統之前就已經被清洗和轉換好了。優點是目標系統中的資料品質有保障，缺點是轉換邏輯與擷取綁在一起，彈性較低。

ELT（Extract-Load-Transform）

先擷取、先載入、再轉換。現代資料湖架構的主流。

把原始資料先搬進來（通常是資料湖或雲端倉儲），再利用目標系統的計算能力做轉換。優點是保留原始資料、轉換邏輯可以隨時調整。

如何選擇？

  • 資料量大、儲存便宜 → ELT（利用雲端倉儲的計算能力）
  • 資料品質要求極高 → ETL（在入庫前就確保品質）
  • 需要快速迭代分析 → ELT（保留原始資料，後續可以做不同的轉換）

管線設計原則

冪等性（Idempotency）

同一個任務跑多次，結果應該相同。這是管線設計最重要的原則。

實作方式：

  • 寫入前先刪除目標分區的資料
  • 使用 UPSERT 而非 INSERT
  • 用日期分區，每次只處理特定日期的資料

原子性（Atomicity）

一個步驟要麼完全成功，要麼完全回滾。不能有「做了一半」的狀態。

可觀測性

每個步驟都要有日誌和指標：

  • 處理了多少筆資料？
  • 花了多長時間？
  • 輸入和輸出的資料量是否合理？

排程與編排工具

Apache Airflow

目前最主流的管線編排工具。用 Python 定義 DAG（有向無環圖），每個節點是一個任務。

核心概念：

  • DAG：定義任務的依賴關係
  • Operator：執行具體工作的元件（SQL、Python、Bash 等）
  • Sensor：等待某個條件成立再繼續
  • XCom：任務間傳遞小量資料

其他工具

工具特色適用場景

管線監控與告警

管線跑失敗是常態，重要的是你能多快知道、多快修復。

該監控什麼？

  • 執行狀態：成功/失敗/重試
  • 執行時間：異常延長可能代表上游資料量暴增或系統資源不足
  • 資料量：突然歸零或暴增都是異常信號
  • 資料品質：缺失率、異常值比例

SLA（服務水準協議）

設定管線的完成期限。例如「每天上午 9 點前完成」。超時要自動告警。

常見陷阱

  1. 時區問題：UTC 還是當地時間？管線排程和資料中的時間戳要統一
  2. 上游異動：上游資料表結構變了，你的管線就壞了。要設防禦性檢查
  3. 回填（Backfill）：修改邏輯後需要重新處理歷史資料。設計管線時要考慮這個需求
  4. 資源競爭：很多管線同時跑，搶佔叢集資源。要設定優先級和資源配額

實務心法

  1. 冪等性是第一優先——任何管線都應該能安全地重跑
  2. 先小後大：先用小量資料驗證邏輯，再處理全量
  3. 監控比開發更重要：一條沒有監控的管線遲早會出問題
  4. 文件化：記錄每條管線的目的、輸入輸出、負責人

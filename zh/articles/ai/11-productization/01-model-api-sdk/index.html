訓練好的模型只是半成品。要讓其他系統和團隊能使用它，你需要把它包裝成易用的 API 或 SDK。

模型 API 設計

REST API 設計原則

端點設計：

  • 預測端點：POST /v1/predictions
  • 批次預測：POST /v1/predictions/batch
  • 模型資訊：GET /v1/models/{model_id}
  • 健康檢查：GET /v1/health

請求格式：

  • 清楚定義輸入 Schema
  • 支援多種輸入格式（JSON、base64 圖片等）
  • 設定輸入大小限制

回應格式：

  • 預測結果
  • 信心度分數
  • 模型版本資訊
  • 請求 ID（方便除錯）

版本控制

API 版本控制讓你能更新模型而不影響現有使用者：

  • URL 版本：/v1/, /v2/
  • Header 版本：Accept: application/vnd.api+v2+json

非同步 API

對於耗時較長的預測（如影片分析、大批次處理）：

  1. 客戶端提交請求，獲得 Job ID
  2. 服務端在背景處理
  3. 客戶端用 Job ID 查詢結果

串流 API

對於生成式 AI（如 LLM）：

  • 使用 Server-Sent Events（SSE）或 WebSocket
  • 逐步返回生成的結果
  • 降低用戶感知的延遲

SDK 設計

為什麼需要 SDK？

直接呼叫 API 太底層了。SDK 提供更好的開發體驗：

  • 類型安全
  • 自動重試和錯誤處理
  • 連接管理
  • 語言慣用的介面

SDK 設計原則

簡單至上：最常見的用例應該一行程式碼就能完成。

一致性：命名、參數順序、錯誤處理方式要一致。

可擴展性：進階用戶能自訂行為（超時、重試策略等）。

良好的錯誤訊息：不要返回 HTTP 500，要告訴用戶哪裡出了問題、如何修正。

API 品質保證

速率限制

保護服務不被過度使用：

  • 每秒請求數限制
  • 每日配額
  • 按用戶或 API Key 分別限制

認證與授權

  • API Key：最簡單的方式
  • OAuth 2.0：更安全，適合第三方整合
  • JWT：無狀態認證

監控

  • 請求量和延遲趨勢
  • 錯誤率和錯誤類型分佈
  • 各版本的使用比例
  • 客戶端版本分佈

SLA 定義

明確承諾：

  • 可用性（如 99.9%）
  • 延遲上限（如 P99 < 200ms）
  • 吞吐量保證
  • 資料保留政策

實務心法

  1. API 設計要以使用者為中心——不是模型為中心
  2. 版本控制從第一天開始——否則未來會很痛苦
  3. 非同步 API 適合重計算——不要讓客戶端等待
  4. SDK 要有完整的使用範例和文件——最好的 SDK 是不需要看文件就能用的

模型打包與匯出

將訓練好的模型打包成可部署的形式：序列化、格式轉換與環境封裝

模型訓練完成後，它只是一堆數字（權重）存在記憶體裡。要把它部署到生產環境，你需要把它「打包」成一個可攜帶、可運行的形式。

模型序列化

序列化（Serialization）是把模型從記憶體狀態轉換為可儲存、可傳輸的格式。

框架原生格式

PyTorch：

  • torch.save(model.state_dict())：只儲存權重（推薦）。載入時需要先建立模型結構。
  • torch.save(model)：儲存整個模型。方便但有 Python 版本和類別路徑的依賴。

TensorFlow/Keras：

  • SavedModel 格式：包含計算圖和權重。是 TF Serving 的標準輸入。
  • HDF5 格式：Keras 的傳統格式，逐漸被 SavedModel 取代。

跨框架格式

ONNX（Open Neural Network Exchange）： 微軟推動的開放格式，幾乎所有框架都支援匯出。

好處：

  • 一次訓練，多處部署
  • 可以用 ONNX Runtime 在各種硬體上高效推論
  • 支援模型優化（圖優化、量化等）

TorchScript： PyTorch 的靜態圖格式。可以在沒有 Python 的環境中運行。

兩種建立方式：

  • Tracing：給定輸入，追蹤模型的計算過程
  • Scripting：分析 Python 程式碼，轉換為 TorchScript

Tracing 不支援控制流程（if-else），Scripting 有 Python 子集的限制。

傳統 ML 模型

  • Scikit-learn：用 joblib 或 pickle 序列化
  • XGBoost/LightGBM：框架自帶的 save/load
  • PMML：老牌跨平台格式，適合企業環境

模型優化

部署前的模型優化能顯著降低推論成本。

量化（Quantization）

把模型權重從 float32 降低到 int8 或 float16。

方式原理效能影響
訓練後量化訓練完再量化可能有少量精度損失
量化感知訓練訓練時模擬量化效果精度損失更小
動態量化只量化權重，啟動值保持 float最安全的選擇

剪枝（Pruning）

移除不重要的權重或神經元。

  • 非結構化剪枝：移除個別權重（變成稀疏矩陣）
  • 結構化剪枝：移除整個神經元或通道（更容易加速）

知識蒸餾（Knowledge Distillation）

用大模型（教師）的輸出來訓練小模型（學生）。學生模型學習的不只是正確答案，還有教師模型的「軟標籤」（各類別的機率分佈）。

環境封裝

Docker 容器

把模型、推論程式碼和所有依賴打包成 Docker 映像。

好處：

  • 環境完全一致——開發、測試、生產用同一個映像
  • 可以在任何支援 Docker 的平台上運行
  • 容易水平擴展

模型伺服器

不用自己寫 HTTP 伺服器，直接用現成的模型伺服器：

伺服器適用場景
TorchServePyTorch 模型
TF ServingTensorFlow 模型
Triton多框架，高吞吐量
BentoML自定義推論邏輯

實務心法

  1. 開發用框架原生格式，部署用 ONNX 或 TorchScript
  2. 量化幾乎免費——先試試 int8 量化，大多數模型精度損失可忽略
  3. Docker 是部署的黃金標準——不要讓「環境問題」成為上線的障礙
  4. 部署前一定要做推論效能測試——不只看準確率，也要看延遲和吞吐量

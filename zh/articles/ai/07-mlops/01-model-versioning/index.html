你團隊裡有人訓練了一個新模型，效果很好。但三個月後，沒有人記得它是用什麼資料、什麼參數訓練的。模型版本控制就是為了避免這種混亂。

為什麼模型版本控制比程式碼更複雜？

一個 ML 模型的「版本」不只是程式碼。它是程式碼 + 資料 + 超參數 + 環境 + 隨機種子的組合。少了任何一個，你就無法重現這個模型。

這意味著：

  • Git 只能管理程式碼部分
  • 資料需要獨立的版本控制
  • 模型的二進位檔案（權重）需要額外儲存
  • 訓練環境需要被記錄

模型註冊中心（Model Registry）

模型註冊中心是模型版本控制的核心組件。它維護一個模型的完整履歷。

該記錄什麼？

每個模型版本應該包含：

  • 模型本體：權重檔案、模型定義
  • 元資料：訓練日期、訓練者、訓練時間
  • 來源：程式碼版本、資料版本、超參數
  • 效能指標：在各測試集上的表現
  • 狀態：開發中 → 驗證中 → 生產中 → 退役

模型生命週期狀態

狀態說明
Staging新訓練的模型，待驗證
Production已驗證，正在生產環境服務
Archived舊版本，保留供回溯

工具選擇

MLflow Model Registry：最主流的開源方案。與 MLflow 實驗追蹤無縫整合。

SageMaker Model Registry：AWS 原生方案。與 SageMaker 管線深度整合。

Vertex AI Model Registry：Google Cloud 原生方案。

自建方案：用 S3/GCS + 資料庫也可以做到，但要自己處理很多細節。

打包最佳實踐

一個模型包應該包含：

  • 模型權重
  • 預處理邏輯
  • 後處理邏輯
  • 輸入輸出的 Schema
  • 依賴清單

MLflow 的 Model Flavor 機制提供了很好的打包標準。

環境管理

模型的行為取決於它的依賴環境（Python 版本、庫版本、CUDA 版本等）。

Conda 環境：記錄完整的依賴樹。MLflow 自動生成 conda.yaml。

Docker 映像：最可靠的環境復現方式。把模型和其完整環境打包成容器。

虛擬環境 + requirements.txt：輕量級方案，但可能遺漏系統級依賴。

版本命名策略

語義化版本號

v1.2.3：

  • Major（1）：模型架構變更或不相容的變化
  • Minor（2）：新特徵加入、效能提升
  • Patch（3）：Bug 修復、小調整

日期 + 序號

model-20250101-001：簡單直接，適合快速迭代的場景。

實務心法

  1. 每個上線的模型都必須在 Registry 中有記錄——沒有例外
  2. 自動化記錄勝過手動記錄——把版本控制整合到訓練管線中
  3. 保留足夠的歷史版本——你永遠不知道什麼時候需要回滾
  4. 用 Docker 確保環境一致性——「在我機器上可以跑」不是藉口

深度學習的訓練速度，很大程度上取決於你對 GPU 的利用效率。從單 GPU 的效能優化到多 GPU 的並行訓練，是每個深度學習工程師必須掌握的技能。

GPU 計算的基礎

為什麼 GPU 比 CPU 快？

CPU 擅長處理複雜的邏輯運算，有強大的單核效能和分支預測能力。

GPU 擅長大量簡單運算的並行處理。一個 GPU 有數千個小核心，適合矩陣乘法這種「對很多數字做相同操作」的計算。

而深度學習的核心就是矩陣乘法，所以 GPU 完美契合。

GPU 記憶體管理

GPU 記憶體（VRAM）是最稀缺的資源。訓練中的記憶體佔用包括：

  • 模型參數：模型本身的權重
  • 梯度：與參數同等大小
  • 優化器狀態：Adam 需要存儲一階和二階動量，等於 2 倍參數量
  • 啟動值：前向傳播中間結果，需要保留以計算梯度
  • 資料批次：當前的輸入資料

一個粗略的估算：訓練一個 N 參數的模型（用 Adam），大約需要 16N bytes 的 GPU 記憶體。

減少記憶體使用

混合精度訓練（Mixed Precision）：用 float16 計算前向和反向傳播，用 float32 儲存參數。記憶體減半，速度提升 2-3 倍。

梯度檢查點（Gradient Checkpointing）：不儲存所有啟動值，需要時重新計算。用計算換記憶體。

梯度累積：用小批次計算梯度，累積 N 次後才更新參數。等效於 N 倍的批次大小。

單 GPU 效能優化

DataLoader 優化

  • 使用多程序資料載入（num_workers > 0）
  • 啟用 pin_memory 加速 CPU-GPU 資料傳輸
  • 預處理和增強放在 DataLoader 中做，利用 CPU 閒置時間

避免 CPU-GPU 同步

每次從 GPU 取回資料（如列印 loss 值）都會觸發同步，造成 GPU 等待。

最佳實踐：

  • 累積 loss 在 GPU 上，每 N 步才取回一次
  • 不要在每步都呼叫 .item() 或 .numpy()

使用 torch.compile（PyTorch 2.0+）

自動將模型圖優化並編譯成更高效的核心。通常能帶來 20-40% 的加速。

多 GPU 並行策略

資料並行（Data Parallelism）

最常用的策略。每個 GPU 持有完整的模型副本，處理不同的資料批次。前向傳播後收集梯度，同步更新參數。

DDP（Distributed Data Parallel）：PyTorch 推薦的實作。使用 AllReduce 操作同步梯度，效率遠高於舊的 DataParallel。

模型並行（Model Parallelism）

當模型太大，一個 GPU 放不下時，把模型切成多段放在不同的 GPU 上。

問題：GPU 之間有資料依賴，一個 GPU 在計算時其他 GPU 在等待。解決方案是管線並行（Pipeline Parallelism）——把不同的小批次交錯在不同階段。

FSDP（Fully Sharded Data Parallelism）

結合資料並行和模型並行的優點。每個 GPU 只儲存模型的一個分片，需要時才收集完整參數。大幅降低每個 GPU 的記憶體需求。

Tensor Parallelism

把單個矩陣運算分佈到多個 GPU 上。適合非常大的模型中的大型矩陣乘法。

GPU 訓練的常見問題

OOM（Out of Memory）：

  • 減小批次大小
  • 啟用混合精度訓練
  • 啟用梯度檢查點
  • 減小模型大小

訓練速度不如預期：

  • 檢查 DataLoader 是否是瓶頸（CPU 來不及準備資料）
  • 檢查是否有不必要的 CPU-GPU 同步
  • 確認混合精度是否正確啟用

多 GPU 線性加速未達到：

  • 通訊開銷：GPU 之間同步梯度需要時間
  • 批次大小限制：總批次太大可能影響收斂
  • 資料載入瓶頸：多 GPU 需要更快的資料載入

實務心法

  1. 混合精度訓練幾乎沒有缺點——永遠啟用
  2. DDP 是多 GPU 訓練的首選，除非模型太大需要 FSDP
  3. 先在單 GPU 上優化效能，再擴展到多 GPU
  4. 監控 GPU 利用率——如果低於 80%，通常有優化空間

AWS 雲端的歷史與現況

歷史發展

  • 2002 年：AWS 最初在 Amazon 內部推出
    • Amazon 發現 IT 基礎設施可被「外部化」
    • 將自家核心能力（大規模 IT 基礎設施）轉為對外服務
  • 2004 年：首次公開服務，推出 Amazon SQS
  • 2006 年：服務大幅擴展，正式推出 SQS、S3、EC2
  • 之後：從美國擴展至全球（如歐洲）
  • 現今：大量知名企業與組織使用 AWS（Dropbox、Netflix、Airbnb、NASA 等）

市場地位

  • 在 Gartner 魔力象限（Magic Quadrant）中長期為領導者
  • 2023 年營收：約 900 億美元
  • 2024 Q1 市占率：約 31%（微軟 Azure 約 25%，排名第二）
  • 連續 13 年為雲端市場領導者
  • 超過 100 萬名活躍用戶
  • 學習 AWS 對雲端職涯具高度價值

常見應用類型

  • 幾乎可建構「任何類型」的應用
  • 支援高度可擴展、複雜的系統
  • 適用於各行各業
  • 常見使用情境：
    • 企業 IT 系統上雲
    • 備份與儲存
    • 大數據分析
    • 網站託管
    • 行動 / 社群應用後端
    • 遊戲伺服器全面雲端化
  • 代表客戶：Netflix、McDonald’s、21st Century Fox、Activision

AWS 全球基礎架構

核心組成

AWS 為全球化服務，核心組成包括：

  • Regions（區域）：地理位置分散的區域
  • Availability Zones（可用區）：每個區域內的獨立資料中心
  • Data Centers（資料中心）：實體基礎設施
  • Edge Locations / Points of Presence（邊緣節點）：全球分布的邊緣位置

各 Region 透過 AWS 私有網路互相連接，可利用 AWS 基礎設施快速打造全球化應用。

AWS Region

  • AWS Region 是 AWS 在全球設立的實體地理區域，每個 Region 由數個資料中心群組組成

  • 使用大多數 AWS 服務（如 EC2、S3、Lambda）前都需先選擇區域

  • 有些是 全球性服務（如 IAM、CloudFront、Route 53、AWS Organizations）無需選擇區域

  • 每個 Region 包含多個可用區 AZ（Availability Zone）

  • 分布於全球各地（美國、歐洲、亞洲、南美、非洲等）

如何選擇 Region（考試重點）

  • 法規與合規（Compliance）：資料需留在特定國家（如法國）
  • 延遲（Latency）：服務部署應接近主要使用者
  • 服務可用性：並非每個 Region 都有所有 AWS 服務
  • 價格（Pricing）：不同 Region 價格不同

AZ (Availability Zone)

  • AZ 是 AWS Region 中的一個或多個獨立的資料中心群組，具備冗餘電力、網路與連接
  • 每個 Region 通常有 3 個 AZ（最少 3、最多 6）
    • 例如：Sydney（ap-southeast-2）有 ap-southeast-2a、ap-southeast-2b、ap-southeast-2c
  • 每個 AZ：
    • 由 一個或多個獨立資料中心組成
    • 具備獨立的電力、網路與連線
  • AZ 之間：
    • 實體隔離
    • 設計為災害不會連鎖影響
    • 透過高速、低延遲網路互連
  • 多個 AZ 組合形成一個 Region
  • 可透過跨 AZ 部署應用來達成高可用性（HA）、容錯性與低延遲
  • 若需部署 6 台 EC2 並考慮容錯，做法如：
    • 在 3 個 AZ 中各放 2 台
    • 或在任兩個 AZ 中各放 6 台，確保單一 AZ 故障時仍正常運作

Edge Locations / Points of Presence

  • AWS 在全球擁有：
    • 400+ 個節點
    • 分布於 90 座城市、40 個國家
  • 主要用途：
    • 內容傳遞
    • 降低使用者延遲
  • 常見於 CDN 與邊緣服務（如 CloudFront）

AWS 服務的作用範圍

全球性服務（Global Services）

  • IAM
  • Route 53
  • CloudFront
  • WAF

區域性服務（Region Scoped）

  • EC2
  • Elastic Beanstalk
  • Lambda
  • Rekognition

AWS Organizations

  • 全域服務，用於集中管理多個 AWS 帳號與成本
  • 支援跨帳號使用量合併計費（Consolidated Billing）
  • 可依部門、環境分群 OU（Organization Unit）
  • 有一個主帳號（Master），其餘為成員帳號（Member）
  • 可針對 OU 或帳號套用 SCP（Service Control Policy）限制資源操作範圍
  • SCP 的 Deny 會蓋過 Allow，主帳號不受 SCP 限制
  • 可使用 RAM（Resource Access Manager）跨帳號共用資源（如 Transit Gateway、子網、Route 53 規則）
  • 合併組織：將成員帳號退出舊組織並刪除，再加入新組織
  • AWS Control Tower 整合 Organizations 提供 Landing Zone 快速建立帳號架構與最佳實踐設定

多環境策略建議

帳號分離策略

環境用途特性
Development開發測試寬鬆權限、低成本資源
Staging預發布環境模擬生產環境
Production正式環境嚴格權限、高可用性

最佳實踐

  1. 使用 Organizations 管理多帳號：分離環境、部門或專案
  2. 實施 SCP 限制：防止意外操作影響生產環境
  3. 跨 AZ 部署：確保高可用性
  4. 選擇適當 Region：考量延遲、合規、成本等因素

概述

  • 為 EC2 提供持久性區塊儲存（EBS Volume）
  • 一次只能掛載至單一 EC2，可卸載並掛載至同 AZ 其他 EC2
  • 一台 EC2 可掛多顆 EBS
  • 資料會保留在磁碟中

Instance Store

  • 與 EC2 實體主機綁定的區塊型暫存儲存裝置（Ephemeral）
  • 啟動時綁定，不可動態掛載
  • 停止或終止即清空資料
  • 重開機資料不會消失
  • 提供極低延遲與高隨機 I/O 效能

EBS Volume 類型

SSD 類型（支援開機磁碟）

類型說明適用場景最高 IOPS
gp2通用型 SSD開機磁碟、開發測試16,000
gp3通用型 SSD（新版）開機磁碟、開發測試16,000
io1高效能 SSD關鍵應用、大型資料庫64,000
io2高效能 SSD（新版）關鍵應用、大型資料庫64,000

HDD 類型（不支援開機磁碟）

類型說明適用場景
st1吞吐最佳化 HDD大數據、日誌、資料倉儲
sc1冷存取 HDD最低成本的大型冷資料

Nitro 架構

  • 使用 Nitro 架構可達到最高 64,000 EBS IOPS
  • 非 Nitro 僅 32,000 IOPS

EBS Snapshot

  • 指定時間點的備份
  • 可跨區建立 Volume
  • 增量備份，只儲存變更的區塊

EBS 加密

加密特性

  1. 儲存資料全程加密
  2. 與 EC2 傳輸中的資料加密
  3. Snapshot 自動加密
  4. 從加密快照建立的 Volume 自動加密
  5. 從未加密快照建立 Volume 時可選擇加密

加密未加密的 EBS

  1. 建立 Snapshot
  2. 複製並加密此快照
  3. 從加密快照還原為新 Volume

EBS 操作

線上修改

  • 支援線上修改容量、IOPS、類型
  • 不中斷運作

Multi-Attach

  • io1/io2 支援 Multi-Attach
  • 可同時掛載至多個 EC2（同一 AZ）
  • 需要叢集感知的檔案系統

EBS Volume 類型（2024 更新）

gp3（推薦）

  • 基準 3,000 IOPS 和 125 MiB/s 吞吐
  • IOPS 和吞吐可獨立調整
  • 比 gp2 更具成本效益

io2 Block Express

  • 最高 256,000 IOPS
  • 最高 4,000 MiB/s 吞吐
  • 適用於最嚴苛的工作負載
  • 需要 Nitro 實例

EBS Snapshots Archive

  • 將不常存取的快照移至歸檔層
  • 成本降低 75%
  • 還原時間 24-72 小時

RAID 設定

RAID 類型說明用途
RAID 0條帶化（併聯）效能提升
RAID 1鏡像容錯備援

EFS（Elastic File System）

  • POSIX 相容的檔案系統，支援共享與鎖定
  • 可自動擴展儲存空間
  • 高可用性：跨多個 AZ 儲存資料
  • 使用 NFS 協定，支援同時掛載至上千台 EC2

支援的服務

  • EC2（Linux）
  • ECS
  • EKS
  • Lambda

效能模式

模式說明適用場景
General Purpose低延遲一般用途
Max I/O高併發吞吐但延遲略高分析/媒體處理

EFS 儲存類別

類別說明
Standard跨多個 AZ 儲存
One Zone單一 AZ 儲存，成本較低
Infrequent Access (IA)不常存取的資料
One Zone-IA單一 AZ 不常存取

EFS Replication（複製）

  • 自動將資料複製到另一個區域
  • 支援災難復原和跨區域存取
  • RPO 約 15 分鐘

應用

  • 跨 AZ 共用檔案
  • 軟體更新
  • 影像等資料

FSx 系列

FSx for Windows

  • SMB 協定與 NTFS 檔案系統
  • 支援 Microsoft AD 整合、ACL 與使用者配額

FSx for Lustre

  • 高效能並行 Linux 檔案系統（POSIX 相容）
  • 延遲 <1ms，具高吞吐、可達百萬 IOPS
  • 適用於 ML、HPC、影片處理、EDA 等場景
  • 可與 S3 整合，Lustre 當熱儲存，S3 當冷儲存

部署模式

模式說明
Scratch短期，不持久
Persistent長期，持久儲存

儲存服務比較

服務類型協定適用場景
EBS區塊儲存-EC2 磁碟
EFS檔案儲存NFSLinux 共享檔案
FSx for Windows檔案儲存SMBWindows 共享檔案
FSx for Lustre檔案儲存LustreHPC、ML
S3物件儲存HTTP靜態內容、備份

最佳實踐

  1. 選擇適當的 EBS 類型：根據 IOPS 和吞吐需求
  2. 定期建立 Snapshot：備份重要資料
  3. 使用 EFS 共享檔案：多 EC2 需要共享時
  4. 加密所有 EBS：保護敏感資料
  5. 監控 IOPS 使用：避免效能瓶頸
  6. 使用 gp3 取代 gp2：更好的性價比
  7. 使用 EFS One Zone：非關鍵資料可降低成本
  8. 使用 Snapshots Archive：長期保存的快照

儲存基礎概念

儲存類型

類型說明
Direct（本地）儲存EC2 Host 上的儲存
Network 儲存透過網路傳輸（EBS）
Ephemeral 儲存暫時性儲存
Persistent 儲存持久性儲存，超越實例生命週期

儲存架構

架構可掛載可開機說明
Block 儲存✅✅以區塊集合呈現，無結構
File 儲存✅❌有檔案結構的共享
Object 儲存❌❌物件集合，扁平結構（S3）

儲存效能

  • IO（區塊大小）：資料單位大小
  • IOPS：每秒輸入/輸出操作數
  • Throughput（MB/s）：實際傳輸速度

範例：Block size 16KB × IOPS 100 = 1.6 MB/s

EBS 深入解析

基本概念

  • 區塊儲存：原始磁碟配置（Volume），可使用 KMS 加密
  • 實例看到區塊裝置，建立檔案系統（ext3/4, xfs）
  • 儲存在單一 AZ（AZ 韌性）
  • 透過儲存網路附加到 EC2 實例
  • 可卸載和重新附加，不與實例生命週期綁定（持久性）
  • Snapshot 備份到 S3，可從快照建立 Volume（跨 AZ 遷移）
  • 按 GB/月 計費（某些情況還有效能費用）

EBS Volume 類型詳解

GP2

  • 1GB 到 16TB
  • 1 IO Credit = 16KB 資料塊
  • IO Credit Bucket 容量 540 萬 IO Credits
  • 最小 100 IO Credits/秒（任何大小）
  • 超過此限制，每 GB 3 IO Credits/秒（基準效能）
  • 可爆發到 3000 IOPS（消耗 Bucket）
  • Bucket 啟動時是滿的
  • 超過 1TB 時基準超過爆發，不使用 Credit 系統
  • GP2 最高 16,000 IOPS

GP3

  • 移除 Credit Bucket 架構
  • 基準 3000 IOPS
  • 基準 125 MiB/s
  • 比 GP2 便宜 20%
  • 額外付費可達 16,000 IOPS 或 1000 MiB/s
  • 最大吞吐是 GP2 的 4 倍（1000 vs 250 MiB/s）

Provisioned IOPS SSD（io1/io2）

  • IOPS 可獨立於大小調整
  • 一致低延遲和抖動
  • 上限：
    • 64,000 IOPS/Volume（io1/io2）
    • 256,000 IOPS/Volume（Block Express）
    • 1000 MB/s 吞吐（io1/io2）
    • 4000 MB/s 吞吐（Block Express）
  • 容量：4GB-16TB（io1/io2），4GB-64TB（Block Express）

HDD 類型

類型st1（吞吐優化）sc1（冷儲存）
容量125GB - 16TB125GB - 16TB
最高 IOPS500（1MB 區塊）250（1MB 區塊）
最高吞吐500 MB/s250 MB/s
基準40 MB/s/TB12 MB/s/TB
爆發250 MB/s/TB80 MB/s/TB
適用大數據、日誌、串流冷資料、低存取

Instance Store Volumes

概述

  • 區塊儲存裝置，實體連接到 EC2 Host
  • 該 Host 上的實例可存取
  • AWS 最高儲存效能
  • 包含在實例價格中
  • 必須在啟動時附加，無法之後新增

特性

  • D3 = 4.6 GB/s 吞吐
  • I3 = 16 GB/s 連續吞吐
  • 比 EBS 更高 IOPS 和吞吐

重點

  • EC2 Host 本地儲存
  • 僅能在啟動時新增
  • 實例移動、調整大小或硬體故障時資料遺失
  • 高效能但高風險
  • 暫時性：不用於持久性資料

EBS vs Instance Store 選擇

選 EBS

  • 需要持久性
  • 需要韌性
  • 儲存獨立於實例生命週期
  • 應用內建複製的韌性
  • 高效能需求

選 Instance Store

  • 應用內建複製的韌性
  • 高效能需求
  • 超高效能需求
  • 成本考量（通常包含在實例價格）

效能選擇指南

  • 便宜 = ST1 或 SC1
  • 吞吐、串流 = ST1
  • 開機 = 不能用 ST1 或 SC1
  • GP2/3 = 最高 16,000 IOPS
  • IO1/2 = 最高 64,000 IOPS（256,000 Block Express）
  • RAID0 + EBS = 最高 260,000 IOPS
  • 超過 260,000 IOPS = Instance Store

EBS Snapshots 詳解

概述

  • Snapshots 是 EBS Volume 消耗資料的備份，儲存在 S3
  • 增量備份，首次完整備份，之後只備份差異
  • 可用於跨 AZ 或跨區域遷移

特性

  • 首次是使用資料的完整複製
  • 後續是增量（只儲存差異）
  • 可從 Snapshot 建立 Volume
  • 可複製到其他區域
  • 停止和啟動 EC2 會移到其他 Host，會遺失 Instance Store 資料

效能

  • 新 EBS Volume = 立即完整效能
  • 快照還原是延遲載入（漸進取得）
  • 請求的區塊會立即取得
  • 可強制讀取所有資料以立即載入
  • Fast Snapshot Restore（FSR）= 立即還原
    • 每區域最多 50 個快照
    • 設定在 Snap 和 AZ 上

計費

  • 按 GB/月計費
  • 計費使用的資料，非配置的

EBS 加密詳解

概述

  • 預設無加密，增加風險
  • 資料只以加密形式存在於磁碟上
  • 明文資料只存在於 EC2 Host 記憶體中
  • 使用 KMS Keys（aws/ebs 或客戶管理）

重點

  • 帳號可設定預設加密（預設 KMS Key）
  • 或選擇 KMS Key
  • 每個 Volume 使用 1 個唯一 DEK（資料加密金鑰）
  • Snapshots 和後續 Volumes 使用相同 DEK
  • 無法將加密 Volume 改為未加密
  • OS 不知道加密：無效能損失

ENI、實例 IP 和 DNS

ENI（Elastic Network Interface）

  • 每個 EC2 實例至少有一個 ENI（必須在同 AZ）
  • Security Group 是附加在 ENI 上，不是實例
  • ENI 有：
    • MAC Address
    • 私有 IPv4 IP → DNS（ip-10-16-0-10.ec2.internal）
    • 0 或多個次要 IP
    • 0 或 1 個公有 IPv4
    • 每個私有 IPv4 可有 1 個彈性 IP
    • 0 或多個 IPv6
    • Security Groups
    • 來源/目的地檢查

重點

  • 次要 ENI + MAC = 授權（在實例間移動授權）
  • 多宿主（多子網）管理和資料分離
  • 不同 ENI 不同 SG = 更精細的安全控制
  • OS 看不到公有 IPv4
  • 停止和啟動 = IP 變更
  • 公有 DNS = VPC 內解析為私有 IP，其他地方解析為公有 IP

概述

  • AWS 託管的 Serverless NoSQL 資料庫
  • 適合儲存 JSON 結構
  • 單位毫秒等級延遲
  • 適合會話資料與即時查詢

容量模式

模式說明適用場景
Provisioned預先配置讀寫容量單位可預測的工作負載
On-demand按請求計費不可預測的工作負載

DAX（DynamoDB Accelerator）

  • 記憶體內快取服務
  • 加速查詢並降低熱點負載
  • 微秒等級延遲
  • 與 DynamoDB API 相容

DynamoDB Streams

  • 將資料異動事件串流
  • 可觸發 Lambda 函數
  • 保留 24 小時
  • 使用案例：新增使用者即寄送歡迎信

Global Table

  • 支援多區域同步讀寫
  • 需先啟用 Streams
  • Active-Active 複製
  • 適用於全球應用

資料遷移

可透過 Amazon DMS 將以下來源遷移至 DynamoDB：

  • MongoDB
  • Oracle
  • MySQL
  • S3

其他 NoSQL 相關服務

ElastiCache

託管型快取服務，支援 Redis 與 Memcached。

Redis

特性說明
Multi-AZ支援自動故障轉移
驗證支援 Redis Auth
合規支援 HIPAA
持久化支援

Memcached

特性說明
適用場景加速動態網站應用
合規不支援 HIPAA
持久化不支援

Neptune

  • AWS 託管的圖形資料庫
  • 適合社群關係、知識圖譜等高度關聯性資料

ElasticSearch（OpenSearch）

  • 全託管搜尋與分析服務
  • 可整合 Kinesis Firehose、IoT、CloudWatch Logs
  • 適用於模糊搜尋、全文索引、日誌分析

DynamoDB 進階功能

索引類型

類型說明
Local Secondary Index (LSI)與基礎表相同的分區鍵，不同排序鍵
Global Secondary Index (GSI)可使用不同的分區鍵和排序鍵

讀取一致性

類型說明
Eventually Consistent最終一致性，延遲較低
Strongly Consistent強一致性，保證讀取最新資料

Amazon Redshift

  • PB 級資料倉儲服務
  • 使用 MPP（大規模平行處理）架構
  • 支援欄式儲存，優化分析查詢效能
  • 與標準 SQL 相容

Redshift 功能

功能說明
Concurrency Scaling自動擴展以處理並發查詢
Materialized Views預先計算和儲存複雜查詢結果
Redshift ML在 Redshift 中建立和部署 ML 模型
Redshift Serverless無需管理叢集即可運行資料倉儲

使用場景

  • 商業智慧分析
  • 大數據分析
  • 報表和儀表板
  • 機器學習特徵工程

DynamoDB vs RDS 比較

項目DynamoDBRDS
類型NoSQLSQL
結構Key-Value / Document關聯式
SchemaFlexibleFixed
擴展自動水平擴展垂直擴展為主
延遲毫秒級依查詢複雜度
適用場景高吞吐、簡單查詢複雜查詢、交易

最佳實踐

  1. 選擇適當的容量模式：評估工作負載特性
  2. 使用 DAX：需要更低延遲時
  3. 設計好的分區鍵：避免熱點
  4. 啟用 Streams：事件驅動架構
  5. 考慮 Global Table：全球應用需求
  6. 使用 Redshift Serverless：變動分析負載

DynamoDB 架構深入解析

NoSQL 概念

  • 不是一種東西：可以是很多種不同模型
  • 一般沒有 SQL 能力
  • 沒有或彈性的 Schema
  • 關聯式資料庫處理關聯性：資料分散在許多表格中
  • NoSQL 傾向在一個表格中處理：讀取較便宜但寫入較貴

Key-Value 模型

  • 無結構
  • 無關聯性
  • 非常快速
  • 可高度可擴展

Wide Column Store

  • 動態欄位
  • Partition Key（必需）+ 其他鍵

Document 模型

  • 每個 Item 都是 Document
  • 鍵/值：文件可以是 JSON/XML
  • 理想用於訂單、聯絡人、文件結構

DynamoDB 表格設計

表格結構

概念說明
Table基本單位，是項目的集合
Item表格中的一筆資料（類似行）
Attribute項目中的屬性（類似欄位）
Primary Key唯一識別項目的鍵

主鍵類型

類型組成使用場景
Simple (Partition Key)PK唯一值即可識別項目
CompositePK + SK需要組合鍵識別項目

鍵的特性

  • Partition Key（PK）：決定資料儲存位置
  • Sort Key（SK）：可選，允許同一 PK 下有多個項目
  • 每個項目最大 400 KB

容量模式詳解

On-Demand 模式

  • 按請求計費
  • 未知工作負載
  • 不可預測的流量
  • 需要支付使用量

Provisioned 模式

  • 設定 RCU（讀取容量單位）和 WCU（寫入容量單位）
  • 每個操作消耗對應的 RCU/WCU
  • 每個表格有 RCU 和 WCU 突發池（300 秒）

容量單位計算

操作類型容量單位
1 WCU每秒寫入 1 KB
1 RCU（強一致性）每秒讀取 4 KB
1 RCU（最終一致性）每秒讀取 8 KB

操作類型

查詢（Query）

  • 接受單一 PK 值和可選的 SK 或範圍
  • 容量消耗基於返回項目的大小

掃描（Scan）

  • 較不靈活
  • 掃描整個表格後消費容量
  • 效率較低，應盡量避免

一致性模型

最終一致性（Eventually Consistent）

  • 讀取可能不會立即反映最近的寫入
  • 較便宜（消耗一半 RCU）
  • 適用於可接受短暫延遲的讀取

強一致性（Strongly Consistent）

  • 讀取保證返回最新資料
  • 較貴（消耗完整 RCU）
  • 適用於需要即時資料的場景

索引深入解析

Local Secondary Index（LSI）

  • 使用相同的 Partition Key
  • 使用不同的 Sort Key
  • 必須在表格建立時定義
  • 最多 5 個 LSI
  • 與表格共用 RCU/WCU

Global Secondary Index（GSI）

  • 可以有不同的 PK 和 SK
  • 可以隨時建立
  • 最多 20 個 GSI
  • 有自己的 RCU/WCU
  • 最終一致性

DynamoDB Streams 詳解

功能

  • 按時間順序記錄項目變更
  • 24 小時滾動視窗
  • 在 Lambda 中啟用觸發器
  • 報告 INSERT、UPDATE、DELETE

視圖類型

視圖類型包含內容
KEYS_ONLY僅鍵屬性
NEW_IMAGE變更後的整個項目
OLD_IMAGE變更前的整個項目
NEW_AND_OLD_IMAGES變更前後的完整項目

使用案例

  • Lambda 觸發器
  • 分析和報告
  • 資料彙總
  • 跨區域複製

DynamoDB Triggers

架構

DynamoDB → Streams → Lambda → 其他服務/資料庫

特性

  • 動作產生：當 Streams 發送資料給 Lambda
  • Streams 的項目變更會觸發 Lambda
  • 可用於觸發其他操作

Global Tables

概念

  • 多主動複製
  • 表格在多個 AWS 區域中複製
  • 最後寫入者獲勝（用於衝突解決）

要求

  • 必須先啟用 Streams
  • 所有副本需使用相同的表格名稱
  • 需要全域一致的應用邏輯

效能

  • 亞秒級複製
  • 強一致性讀取（同一區域）
  • 最終一致性讀取（跨區域）

DynamoDB Accelerator（DAX）詳解

傳統快取 vs DAX

傳統快取：

  1. 應用程式檢查快取
  2. 快取 Miss → 查詢資料庫
  3. 存入快取
  4. 下次快取 Hit

DAX：

  • SDK 改用 DAX SDK
  • 無需客戶端程式碼
  • DAX 叢集在多個 AZ 內運行
  • 同時快取項目和查詢

DAX vs ElastiCache

比較項目DAXElastiCache
用途DynamoDB 專用通用快取
快取內容項目 + 查詢需自訂
整合DynamoDB API 相容需應用程式整合
適用讀取密集型 DynamoDB需儲存運算結果

TTL（Time To Live）

功能

  • 自動刪除過期項目
  • 以 Epoch 時間（秒）定義
  • 過期後會被刪除

特性

  • 過期項目保留在索引中直到刪除
  • 不消耗 WCU
  • 每個分區每秒可刪除 14000 個項目

ElastiCache 深入解析

架構

  • 記憶體內鍵值存儲
  • 可用於暫時性資料或在 DynamoDB 上快取
  • 需要程式碼變更
